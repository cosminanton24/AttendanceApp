CHAPTER 1 – PROJECT OVERVIEW AND MOTIVATION
================================================================================

1.1  Introduction
-----------------
AttendanceApp is a web-based academic attendance management system designed to
support modern university environments where students and professors interact
digitally around lectures, quizzes, and course participation. The system enables
professors to schedule and manage lectures, start real-time in-class sessions,
require students to sign in using GPS-based geofencing, and issue live quizzes
during those sessions. Students, in turn, follow professors, join ongoing
lectures by submitting their GPS location, answer quiz questions in real time,
and submit their final quiz responses.

The application was built as a full-stack system comprising a RESTful Web API
back end, an MVC server-rendered front end, a PostgreSQL relational database, and
a comprehensive suite of unit and integration tests. The solution is
containerised using Docker so that the database and ancillary services can be
started with a single command.

1.2  Problem Statement
-----------------------
Traditional attendance management in universities relies on paper-based registers
or simple sign-in sheets, which are susceptible to fraud (students signing on
behalf of absent peers), administrative overhead, and the absence of real-time
visibility for professors. Digital solutions that require purpose-built hardware
(NFC badges, biometric scanners) incur high infrastructure costs.

AttendanceApp addresses this by:
  – Using the GPS sensor already present in every smartphone to verify physical
    proximity of a student to the lecture room (within 100 metres).
  – Providing professors with an instant web dashboard showing who is present.
  – Allowing in-lecture quiz assessments whose results and scores are
    automatically computed and made available immediately after submission.
  – Incorporating a social-follow mechanism so students can discover and
    subscribe to professors whose lectures they are interested in.

1.3  Scope of the System
------------------------
The system handles the following principal use-cases:

  1. User registration and authentication (students, professors, admins).
  2. Lecture lifecycle management: scheduling, starting (with GPS), ending,
     cancelling, and deletion.
  3. Attendance tracking: students join in-progress lectures by providing their
     current GPS coordinates; the system rejects a join request if the
     calculated Haversine distance to the lecture location exceeds 100 metres.
  4. Quiz management: professors create quizzes, add questions and multiple-
     choice options, and activate a quiz for a specific live lecture.
  5. Quiz participation: students see the active quiz, answer questions, and
     submit their completed quiz; the system calculates and stores their score.
  6. Social following: students follow professors; followers/following counts are
     visible on profile pages.
  7. Paginated queries with optional filtering across all major entities.

CHAPTER 2 – ARCHITECTURAL DESIGN
================================================================================

2.1  Clean Architecture
------------------------
AttendanceApp follows the Clean Architecture pattern (also known as Onion
Architecture or Hexagonal Architecture) as formulated by Robert C. Martin. The
fundamental principle of Clean Architecture is the Dependency Rule: source code
dependencies always point inwards—towards higher-level policy—and never outward
towards implementation details.

The solution is divided into five deployable / compilable projects plus two test
projects, arranged in the following concentric layers (innermost first):

  +----------------------------------------------------+
  |           4. AttendanceApp.Web  (MVC UI)           |
  |     3. AttendanceApp.Api  (REST Controllers)       |
  |   2. AttendanceApp.Infrastructure  (EF Core, DB)   |
  |   1. AttendanceApp.Application  (Use Cases, CQRS)  |
  |       0. AttendanceApp.Domain  (Entities, Rules)   |
  +----------------------------------------------------+

    AttendanceApp.UnitTests         (tests Domain + Application + Api)
    AttendanceApp.IntegrationTests  (tests Web + Api via in-memory EF)

Dependency arrows:

  Domain        ←  Application
  Application   ←  Infrastructure
  Application   ←  Api
  Domain        ←  Infrastructure
  Domain + Infrastructure + Application + Api  ←  Web

The Domain project has NO external NuGet dependencies whatsoever. The Application
project depends only on the Domain plus three NuGet packages (MediatR,
FluentValidation, Konscious Argon2). Infrastructure is the only project that
knows about Entity Framework Core and PostgreSQL. The API project knows about
ASP.NET Core controllers and Swashbuckle but does not reference Infrastructure
directly—it goes through Application interfaces.

2.2  Project Dependency Graph
------------------------------
The Visual Studio solution (AttendanceApp.sln) registers seven projects:

  AttendanceApp.Domain          – no project references
  AttendanceApp.Application     – → Domain
  AttendanceApp.Infrastructure  – → Application   (and transitively → Domain)
  AttendanceApp.Api             – → Infrastructure (and transitively → everything)
  AttendanceApp.Web             – → Domain, Application, Infrastructure, Api
  AttendanceApp.UnitTests       – → Domain, Application, Api
  AttendanceApp.IntegrationTests– → Api, Web, Infrastructure

2.3  Layer Responsibilities
----------------------------

  Domain Layer
    Defines the core business entities, value objects, domain exceptions, and
    repository interfaces. Contains NO framework or database awareness.

  Application Layer
    Encodes application use-cases using the CQRS pattern via MediatR. Each use
    case is implemented as a Command (mutation) or Query (read), paired with a
    handler and a FluentValidation validator. Cross-cutting concerns (validation
    pipeline, exception conversion) are handled through MediatR pipeline
    behaviors.

  Infrastructure Layer
    Implements the repository interfaces declared in the Domain using Entity
    Framework Core 10 and the Npgsql provider for PostgreSQL. Contains the
    DbContext, all Fluent API entity configurations, custom value converters, and
    EF migration history.

  API Layer (AttendanceApp.Api)
    A minimal ASP.NET Core project that registers Swagger/OpenAPI. Its
    controllers use constructor-injected IMediator to dispatch commands/queries
    and convert Result<T> values to HTTP responses via a shared extension method.

  Web Layer (AttendanceApp.Web)
    An ASP.NET Core MVC project that serves server-rendered Razor views as the
    user-facing front end. It hosts all service registrations (DI container
    setup, JWT bearer auth, EF, MediatR, repositories) and uses the same API
    controllers through the composition root. Static front-end assets (Bootstrap
    5, jQuery, custom CSS and JavaScript) are served from wwwroot.

2.4  Request Flow
------------------
A typical HTTP request follows this pipeline:

  Browser / API Client
      |
      v
  ASP.NET Core Routing  (Program.cs middleware pipeline)
      |
      v
  Authentication Middleware  (JWT cookie/header validation)
      |
      v
  Authorization Middleware   ([Authorize] attribute check)
      |
      v
  Controller Action          (e.g. LectureController.CreateLecture)
      |  extracts authenticated user ID from ClaimsPrincipal
      |  maps request DTO → MediatR command
      v
  IMediator.Send(command)
      |
      v
  MediatR Pipeline
      |  1. ExceptionToResultBehavior  (catches exceptions → Result.Fail)
      |  2. ValidationBehavior         (runs FluentValidation)
      v
  Command / Query Handler   (business logic, calls repository interfaces)
      |
      v
  EF Core Repository        (SQL query generated → Npgsql → PostgreSQL)
      |
      v
  Result<T>  returned up the call stack
      |
      v
  ToActionResult() extension  (maps SuccessKind / ErrorKind → HTTP status code)
      |
      v
  HTTP Response (JSON body or ProblemDetails)

      – Primary constructors (e.g. CreateUserCommandHandler(IUserRepository))

-------------------
  ASP.NET Core 10 (Microsoft.AspNetCore.*)
    The primary server framework. It is used in two modes:
      a) As a Web API backend through attribute-based controllers in
         AttendanceApp.Api that return JSON.
      b) As an MVC application in AttendanceApp.Web that additionally renders
         Razor (.cshtml) views and serves static files.

  Swashbuckle.AspNetCore 10.1.0
    Provides automatic OpenAPI (Swagger) specification generation and the

    referenced by AttendanceApp.Api.

3.3  Mediator / CQRS
---------------------
  MediatR 14.0.0
    MediatR is the central dispatch mechanism for commands and queries. Every
    use-case class implements either IRequest<TResponse> (command or query) and
    is handled by a class implementing IRequestHandler<TRequest, TResponse>.
    MediatR also supports pipeline behaviors (IPipelineBehavior<TRequest,
    TResponse>), which the application uses to add cross-cutting concerns without
    touching individual handlers.

    MediatR is registered against the IAssemblyMarker interface in the
    Application project to locate all handlers automatically at startup:
      builder.Services.AddMediatR(cfg =>
          cfg.RegisterServicesFromAssembly(typeof(IAssemblyMarker).Assembly));

3.4  Validation
----------------
  FluentValidation 12.1.1 and FluentValidation.DependencyInjectionExtensions 12.1.1
    FluentValidation provides a declarative, fluent API for building strongly
    typed validation rules. Each MediatR command/query that requires input
    validation has a companion validator class inheriting AbstractValidator<T>.
    These validators are auto-registered using:
      builder.Services.AddValidatorsFromAssembly(typeof(IAssemblyMarker).Assembly);
    They are invoked by the ValidationBehavior MediatR pipeline behavior.

3.5  Object-Relational Mapping
-------------------------------
  Microsoft.EntityFrameworkCore 10.0.2
    Entity Framework Core is the ORM used throughout the Infrastructure layer.
    The code-first approach is used: entities are defined in the Domain, and
    entity configurations are declared using the Fluent API via IEntityType-
    Configuration<T> classes in the Persistence/Configurations directory.

  Microsoft.EntityFrameworkCore.Relational 10.0.2
    Provides relational-specific features such as HasConversion value converters,

  Npgsql.EntityFrameworkCore.PostgreSQL 10.0.0
    The Npgsql provider connects EF Core to PostgreSQL. It enables PostgreSQL-
    specific features such as the EF.Functions.ILike operator (case-insensitive
    LIKE used for name/email search in the attendee and submission repositories),
    uuid primary key support, timestamp with time zone, and interval types.

  Microsoft.EntityFrameworkCore.Design 10.0.2
    Used at development time for the dotnet-ef CLI to generate and apply
    migrations. Marked as a development-only dependency (PrivateAssets="all").

3.6  Authentication
--------------------
  Microsoft.AspNetCore.Authentication.JwtBearer 9.0.10
    JWT (JSON Web Token) Bearer authentication middleware. The application uses
    HS256 symmetric signing. The secret key is read from the JWT_SECRET_KEY
    environment variable at runtime. The middleware validates token expiry,
    signature, and lifetim. Token delivery uses HTTP-only cookies rather than
    localStorage for improved XSS resistance.

  System.IdentityModel.Tokens.Jwt (via Microsoft.IdentityModel)
    Provides JwtSecurityTokenHandler and JwtSecurityToken used in
    JwtTokenGenerator to build and serialize tokens.

3.7  Password Hashing
----------------------
  Konscious.Security.Cryptography.Argon2 1.3.1
    Argon2id is used for password hashing, as recommended by OWASP for new
    applications. The PasswordHasher utility configures:
      – Memory cost: 128 MB (128 * 1024 KB)
      – Iterations:  3
      – Parallelism: 2 threads
      – Hash size:   32 bytes
      – Salt size:   16 bytes (randomly generated per password via
                     RandomNumberGenerator.GetBytes)
    The final stored string is Base64(salt || hash). Constant-time comparison
    (XOR-reduce) prevents timing attacks during verification.

CHAPTER 4 – DOMAIN LAYER (AttendanceApp.Domain)
================================================================================

4.1  Overview
--------------
The Domain layer is the innermost layer of the architecture. It is a pure class
library (Microsoft.NET.Sdk, target net10.0) with no external NuGet packages.
It defines the business model of the system: entities, aggregate roots, value
objects, enumerations, domain exceptions, guard clauses, and repository
interfaces.

4.2  Common Building Blocks
----------------------------

  4.2.1  Entity<TId>  (Common/Entity.cs)
  ----------------------------------------
  The generic base class for all domain objects that have a persistent identity.

    public abstract class Entity<TId> where TId : notnull
    {
        public TId Id { get; protected set; } = default!;
        protected Entity(TId id) => Id = id;
        protected Entity() { }

        public override bool Equals(object? obj)
            => obj is Entity<TId> other
               && EqualityComparer<TId>.Default.Equals(Id, other.Id);

        public override int GetHashCode() => Id.GetHashCode();
    }

  Identity equality is enforced: two Entity instances are equal if and only if
  their Ids are equal. This implements the DDD (Domain-Driven Design) concept of
  entity identity. The protected no-arg constructor exists solely to satisfy EF
  Core's requirement for a parameterless constructor during object materialisation.

  4.2.2  AggregateRoot<TId>  (Common/AggregateRoot.cs)
  -------------------------------------------------------
  Thin subclass of Entity<TId> marking an entity as an aggregate root—the sole
  entry point through which all invariants of an aggregate are enforced.

    public abstract class AggregateRoot<TId> : Entity<TId> where TId : notnull

  In DDD, aggregate roots are the only objects stored by repositories. The system
  has three aggregate roots: User, Lecture, and Quiz.

  4.2.3  DomainException  (Common/DomainException.cs)
  -----------------------------------------------------
  A sealed exception type thrown exclusively by domain entities and Guard to
  signal business-rule violations. It extends Exception and accepts a message.

    public sealed class DomainException(string message) : Exception(message)

  This type is caught by ExceptionToResultBehavior in the Application layer and
  converted to an Error of kind ErrorKind.Validation, producing an HTTP 400 Bad
  Request response without leaking stack traces.

  4.2.4  Guard  (Common/Guard.cs)
  ---------------------------------
  An internal static utility class providing pre-condition checks used in entity
  constructors and methods. Methods throw DomainException on violation:

    Guard.NotNull(value, name)
      – Throws if value is null.
    Guard.NotNullOrWhiteSpace(value, name, maxLen?)
      – Throws if string is null, empty, whitespace, or exceeds optional maxLen.
    Guard.InRange(value, name, min, max)
      – Throws if integer is outside [min, max].
    Guard.Positive(decimal value, name)
      – Throws if value <= 0.
    Guard.Positive(TimeSpan value, name)
      – Throws if TimeSpan <= TimeSpan.Zero.
    Guard.NotNegative(decimal value, name)
      – Throws if value < 0.
    Guard.NotEmpty(Guid value, name)
      – Throws if Guid == Guid.Empty.
    Guard.NotInPast(DateTime value, name)
      – Throws if DateTime < DateTime.UtcNow.

  The class is marked internal so it cannot be used outside the Domain assembly,
  keeping domain invariant enforcement within the domain itself.

  4.2.5  Location  (Common/Locartion.cs)
  ----------------------------------------
  (Note: the file is named Locartion.cs with a typo; the type is correctly named
  Location.)

  A readonly record struct representing a GPS coordinate triple:

    public readonly record struct Location
    {
        public double Latitude  { get; init; }
        public double Longitude { get; init; }
        public double Accuracy  { get; init; }
    }

  Being a value type (struct) and immutable (readonly record), two Locations with
  identical Latitude, Longitude, and Accuracy are considered the same value.

  The DistanceTo(Location other) method implements the Haversine formula to
  compute the great-circle distance in metres between two geographic coordinates:

    d = 2R * arcsin( sqrt(
          sin²(Δlat/2) + cos(lat1)*cos(lat2)*sin²(Δlon/2)
        ))

  where R = 6,371,000 metres (mean Earth radius).

  ToString() serialises to "lat,lon,accuracy" (comma-separated, invariant
  culture), and static FromString(string) parses this representation back. This
  string form is used by the Infrastructure layer's LocationConverter to
  store Location values in the database as a single VARCHAR column.

4.3  Enumerations
------------------

  4.3.1  LectureStatus  (Enums/LectureStatus.cs)
  ------------------------------------------------
  Represents the lifecycle state of a Lecture entity:

    Scheduled = 0    – The default state when a lecture is created.
    InProgress = 1   – The professor has called Start(), providing a location.
    Ended = 2        – The professor has called End().
    Canceled = 3     – The lecture was cancelled via ChangeStatus.

  Status transitions are enforced by domain methods on Lecture:
    – Start() requires Status == Scheduled; throws DomainException otherwise.
    – End() requires Status == InProgress; throws DomainException otherwise.
    – ChangeStatus() allows any transition without guard (used for admin/cancel).

  4.3.2  UserType  (Enums/UserType.cs)
  --------------------------------------
  Role classification for system users:

    Student   = 0
    Professor = 1
    Admin     = 2

  Business rules depend on UserType. For example, only professors (and admins)
  may create lectures; only students may join lectures.

4.4  Aggregate: Lecture  (Lectures/)
--------------------------------------

  4.4.1  Lecture  (Lectures/Lecture.cs)
  ----------------------------------------
  Aggregate root representing a scheduled academic session.

  Properties:
    Guid         Id            – set in constructor, never regenerated
    Guid         ProfessorId   – immutable reference to creator
    LectureStatus Status       – mutable, transitions via Start/End/ChangeStatus
    string       Name          – up to 256 characters
    string       Description   – free-form text
    DateTime     StartTime     – must be in the future at construction time
    TimeSpan     Duration      – must be positive
    Location?    Location      – null until lecture starts; set by Start()
    DateTime     EndTime       – computed: StartTime + Duration (not mapped to DB)
    IReadOnlyCollection<LectureAttendee> Attendees – read-only projection of
                                private List<LectureAttendee> _attendees

  Constructor validates all fields via Guard and sets the initial status to
  Scheduled with no Location.

  Domain operations:
    void Start(Location location)
      – Validates location is not null.
      – Requires Status == Scheduled; throws DomainException otherwise.
      – Transitions Status → InProgress and records the Location.

    void End()
      – Requires Status == InProgress; throws DomainException otherwise.
      – Transitions Status → Ended.

    void ChangeStatus(LectureStatus status)
      – Unchecked status assignment (used for administrative overrides).

  ORM note: The EndTime property is marked [NotMapped] because it is calculated
  and need not be persisted. The _attendees field is the actual storage; the
  Attendees public property exposes a read-only wrapper.

  4.4.2  LectureAttendee  (Lectures/LectureAttendee.cs)
  -------------------------------------------------------
  An entity (not an aggregate root) representing the recorded attendance of a
  student at a particular lecture.

  Properties:
    Guid     Id              – generated by Guid.NewGuid() in constructor
    Guid     LectureId       – FK to Lecture
    Guid     UserId          – FK to User
    DateTime TimeJoined      – set to DateTime.UtcNow at construction
    Location LocationAtJoin  – GPS coordinates captured at join time

  Constructor validates LectureId and UserId are non-empty via Guard.

  Note that a LectureAttendee is stored as a child of the Lecture aggregate but
  is persisted in a separate table. EF Core follows the Attendees navigation
  property on Lecture to persist/load them.

4.5  Aggregate: User  (Users/)
-------------------------------

  4.5.1  User  (Users/User.cs)
  ------------------------------
  Aggregate root representing a registered system participant.

  Properties:
    Guid     Id       – set in constructor
    UserType Type     – mutable (changeable by admin)
    string   Name     – trimmed; max 100 chars
    string   Email    – trimmed; max 255 chars; unique in database
    string   Password – stores the Argon2id hash, not plaintext

  Domain operations:
    void ChangeName(string name)
    void ChangeEmail(string email)
    void ChangePassword(string password)
    void ChangeType(UserType type)

  All setters validate via Guard. The password field holds the Argon2id hash that
  was computed in the Application layer before the User was constructed.

  Note: The entity also has a private List<Guid> _attendedLectures and its read-
  only counterpart (IReadOnlyCollection<Guid> AttendedLectures). This is a
  vestigial design artefact; the actual attendance relation is handled via the
  LectureAttendee entity and its repository.

  4.5.2  UserFollowing  (Users/UserFollowing.cs)
  ------------------------------------------------
  Entity representing a directed social-follow relationship between two users.

  Properties:
    Guid     Id         – assigned via Guid.NewGuid()
    Guid     FollowerId – the user who follows
    Guid     FollowedId – the user being followed
    DateTime FollowedAt – timestamp of the follow event

  Constructor enforces:
    – Neither ID is empty (Guard.NotEmpty).
    – FollowerId != FollowedId (self-follow throws DomainException).

4.6  Aggregate: Quiz  (Quizzes/)
----------------------------------

  4.6.1  Quiz  (Quizzes/Quiz.cs)
  --------------------------------
  Aggregate root representing a reusable question set created by a professor.

  Properties:
    Guid     Id           – assigned in private constructor
    string   Name         – trimmed; required
    TimeSpan Duration     – must be positive (defines how long students have)
    Guid     ProfessorId  – reference to the creating professor
    DateTime CreatedAtUtc – creation timestamp
    IReadOnlyCollection<QuizQuestion> Questions

  The constructor is private; creation goes through the static factory method
    Quiz.Create(string name, TimeSpan duration, Guid professorId, DateTime now)
  which validates inputs with Guard before calling the constructor. This factory
  pattern ensures the aggregate is always valid upon creation.

  Domain operations:
    void Rename(string newName)
    void ChangeDuration(TimeSpan newDuration)
    QuizQuestion AddQuestion(string text, int order, decimal? points)
    void RemoveQuestion(Guid questionId)

  4.6.2  QuizQuestion  (Quizzes/QuizQuestion.cs)
  -------------------------------------------------
  Entity representing a single question within a quiz.

  Properties:
    Guid     Id      – assigned by static factory Create
    Guid     QuizId  – FK to parent Quiz
    string   Text    – trimmed; max 1024 chars
    int      Order   – display order; must be >= 1
    decimal? Points  – nullable; if set, must be non-negative
    IReadOnlyCollection<QuizOption> Options

  Created exclusively through Quiz.AddQuestion (which calls QuizQuestion.Create
  internally). Domain operations include ChangeText, ChangeOrder, ChangePoints,
  AddOption, RemoveOption, and ClearCorrectOptions.

  4.6.3  QuizOption  (Quizzes/QuizOption.cs)
  --------------------------------------------
  Entity representing a single multiple-choice option for a question.

  Properties:
    Guid   Id         – assigned by Create
    Guid   QuestionId – FK to parent QuizQuestion
    string Text       – trimmed; max 512 chars
    int    Order      – display order; must be >= 1
    bool   IsCorrect  – whether this option is a correct answer

  Created through QuizQuestion.AddOption. Domain operations:
    void ChangeText(string), void ChangeOrder(int),
    void MarkAsCorrect(), void MarkAsIncorrect().

  4.6.4  QuizLecture  (Quizzes/QuizLecture.cs)
  ----------------------------------------------
  Entity representing the activation of a Quiz for a specific live Lecture
  session. It captures the start (CreatedAtUtc) and computed end time
  (EndTimeUtc = CreatedAtUtc + quiz.Duration).

  Key method:
    bool IsActive(DateTime utcNow) => utcNow < EndTimeUtc;
  Used by the repository to determine if students can still answer the quiz.

  Created via static factory QuizLecture.Create.

  4.6.5  UserAnswer  (Quizzes/UserAnswer.cs)
  --------------------------------------------
  Entity recording a student's selection (true/false choice) for a specific
  option in a specific quiz-lecture instance.

  Properties:
    Guid UserId, QuizLectureId, QuestionId, OptionId, bool Choice.

  Students may toggle their choice before final submission; UpdateChoice(bool)
  updates the Choice field.

  4.6.6  UserSubmission  (Quizzes/UserSubmission.cs)
  ----------------------------------------------------
  Entity representing a student's final, scored submission for a quiz-lecture.

  Properties:
    Guid    UserId, QuizLectureId
    bool    Submitted        – always true on creation
    DateTime SubmittedAtUtc
    decimal Score            – computed score
    decimal MaxScore         – maximum achievable score

  Created through static factory UserSubmission.Create. Once created, it is
  immutable (no mutation methods).

4.7  Repository Interfaces  (Repositories/)
--------------------------------------------
The Domain layer defines all repository interfaces, following the Repository
pattern from DDD. Implementations live in the Infrastructure layer.

  IRepository<T> where T : Entity<Guid>
    – Base interface with CRUD operations:
        Task AddAsync(T, CancellationToken)
        Task<T?> GetByIdAsync(Guid, CancellationToken)

CHAPTER 5 – APPLICATION LAYER (AttendanceApp.Application)

5.1  Overview
--------------
The Application layer encodes application-level use-cases using the CQRS
(Command Query Responsibility Segregation) pattern implemented via MediatR. It
references only the Domain layer and its own NuGet packages—it has no knowledge
of EF Core, HTTP, or the database. This ensures the business logic is fully
testable in isolation.

The assembly exports a single public interface IAssemblyMarker (in IAssemblyMaker.cs
— note the slight typo in the file name) whose sole purpose is to serve as a
type anchor for MediatR and FluentValidation assembly scanning:

  public interface IAssemblyMarker { }

5.2  CQRS with MediatR
-----------------------
Every use case is expressed as either a Command (side-effectful mutation) or a
Query (read-only data retrieval):

  Commands implement IRequest<TResponse>:
    – Return types are typically Result<Guid> (for create operations) or Result /
      Result<T> (for other mutations).
    – Examples: CreateUserCommand, CreateLectureCommand, JoinLectureCommand,
      SubmitQuizCommand, SaveUserAnswerCommand, ToggleFollowUserCommand.

  Queries implement IRequest<TResponse>:
    – Return types are typically Result<SomeDto> or Result<PagedDto>.
    – Examples: GetProfessorLecturesQuery, GetStudentLecturesQuery,
      GetQuizzesByProfessorQuery, GetActiveQuizForLectureQuery,
      SearchUsersByNameQuery.

  Handler naming convention:
    CommandHandler or QueryHandler suffix on the implementation class. For
    example, CreateUserCommandHandler : IRequestHandler<CreateUserCommand, Result<Guid>>.

5.4  Selected Feature Walkthroughs
------------------------------------

  5.4.1  CreateUser (Registration)
  ----------------------------------
  Command:  CreateUserCommand(Name, Email, Password, UserType)
  Handler:  CreateUserCommandHandler
    1. Checks for email uniqueness via IUserRepository.GetByEmailAsync.
    2. Calls PasswordHasher.HashPasswordAsync (Argon2id) to produce a hash.
    3. Constructs a new User(Guid.NewGuid(), type, name, email, hash).
    4. Persists via IUserRepository.AddAsync + SaveChangesAsync.
    5. Returns Result<Guid>.Created(newUserId).

  Validator (CreateUserCommandValidator):
    – Name: required, max 100 chars.
    – Email: required, max 255 chars, must be valid email format.
    – Password: required, min 8 chars, max 255 chars.
    – UserType: must be a valid enum value.

  5.4.2  LoginUser (Authentication)
  ------------------------------------
  Command:  LoginUserCommand(Email, Password)
  Handler:  LoginUserCommandHandler
    1. Fetches user by email; throws KeyNotFoundException if not found.
    2. Verifies password via PasswordHasher.VerifyPasswordAsync.
    3. On success, calls JwtTokenGenerator.GenerateToken to produce a JWT.
    4. Returns Result<string>.Ok(jwt).
  The JWT string is returned to the controller, which appends it as an HTTP-only
  cookie (SameSite=Lax, Secure, MaxAge=1 hour).

  5.4.3  CreateLecture
  ---------------------
  Command:  CreateLectureCommand(ProfessorId, Name, Description, StartTime, Duration)
  Handler:  CreateLectureCommandHandler
    1. Verifies the professor user exists.
    2. Checks UserType != Student (only professors/admins may create).
    3. Detects scheduling conflicts via
       ILectureRepository.HasProfessorConflictingLectureAsync.
    4. Constructs Lecture aggregate, persists, returns Result<Guid>.Created.

  5.4.4  JoinLecture (Geofenced Attendance)
  -------------------------------------------
  Command:  JoinLectureCommand(UserId, LectureId, Position)
    – Position is a "lat,lon,accuracy" string parsed to a Location value object.
  Handler:  JoinLectureCommandHandler
    1. Verifies the user exists.
    2. Checks no existing LectureAttendee for this user/lecture pair.
    3. Loads the lecture; verifies Status == InProgress.
    4. Checks the user has no other InProgress lecture already joined.
    5. Verifies user is a Student.
    6. Calls lecture.Location.Value.DistanceTo(Location.FromString(command.Position)).
    7. If distance > 100 metres, throws ValidationException.
    8. Creates LectureAttendee(lectureId, userId, locationAtJoin), persists.
    9. Returns Result.Ok().

  5.4.5  SubmitQuiz
  ------------------
  Command:  SubmitQuizCommand(UserId, QuizLectureId)
  Handler:  SubmitQuizCommandHandler
    1. Verifies no existing UserSubmission for this user/quiz-lecture pair.
    2. Loads the QuizLecture and confirms it is still active.
    3. Loads all UserAnswers for this user/quiz-lecture.
    4. Loads the Quiz with all questions and options.
    5. For each question, sums the points if the user marked the correct option.
    6. Computes MaxScore as the sum of all question points.
    7. Creates UserSubmission.Create(userId, quizLectureId, now, score, max).
    8. Persists and returns Result.Ok().

5.5  Common/Results – The Result Monad
-----------------------------------------
The Application layer introduces two generic railway-oriented programming types
that replace exceptions for control flow between layers:

  Result<T>  – wraps a success value of type T or a list of Error records.
  Result     – non-generic version for operations that return no value.

  Both are readonly structs (value types) for performance (no heap allocation
  for success paths).

  Factory methods for Result<T>:
    Result<T>.Ok(T value)              – HTTP 200
    Result<T>.Created(T value)         – HTTP 201
    Result<T>.NoContent()              – HTTP 204
    Result<T>.OkNullable(T? value)     – OK with potentially null value
    Result<T>.Fail(Error error)        – failure with single error
    Result<T>.Fail(params Error[])     – failure with multiple errors

  Functional combinators on Result<T>:
    Map<TResult>(Func<T, TResult>)     – transforms the success value
    Bind<TResult>(Func<T, Result<TResult>>) – chains fallible operations
    Match<TResult>(onSuccess, onFailure)    – pattern-match on state
    Tap(Action<T>)                     – side-effect on success, returns self
    TapFailure(Action<IReadOnlyList<Error>>) – side-effect on failure, returns self
    TryGetValue(out T)                 – bool + out pattern

  Error record:
    sealed record Error(Code, Message, ErrorKind, Metadata?)
    Factory: Error.Validation, Error.NotFound, Error.Conflict, Error.Unspecified

  ErrorKind enumeration:
    Failure(0), Validation(1), NotFound(2), Conflict(3), Unauthorized(4),
    Forbidden(5)

  SuccessKind enumeration:
    Ok, Created, NoContent

5.6  MediatR Pipeline Behaviors
---------------------------------

  5.6.1  ExceptionToResultBehavior<TRequest, TResponse>
  -------------------------------------------------------
  Wraps every handler execution in a try/catch. The exception mapping table:

    DomainException          → Error.Validation(message)       [no log]
    ValidationException      → Error.Validation(message)       [no log]
    FluentValidation.
      ValidationException    → Error.Validation(message)       [no log]
    KeyNotFoundException      → Error.NotFound(message)        [Warning]
    Any other exception      → Error.Unspecified("unexpected") [Error]

  After mapping, checks if TResponse is Result or Result<T>; if so, calls the
  static Fail factory via reflection and returns the failure without re-throwing.
  This ensures handlers never leak exceptions to controllers.

  5.6.2  ValidationBehavior<TRequest, TResponse>
  ------------------------------------------------
  Before invoking the handler, collects all IValidator<TRequest> instances from
  DI, runs them concurrently (Task.WhenAll), and throws a FluentValidation.
  ValidationException aggregating all failures. This is then caught by
  ExceptionToResultBehavior above, converting it to an HTTP 400 response.

5.7  JWT Utilities (Common/Jwt/)
---------------------------------

  JwtTokenGenerator.GenerateToken(userId, email, userType, expiresInMinutes=60)
    Builds an HS256-signed JWT with claims:
      sub  = userId (string)
      email = user email
      role  = userType.ToString() (e.g. "Student", "Professor")
      jti   = Guid.NewGuid().ToString()
    Signs with the JWT_SECRET_KEY environment variable.

  ClaimsPrincipalExtensions
    Extension methods on ClaimsPrincipal:
      Guid GetUserId()     – reads sub or NameIdentifier claim
      UserType GetUserType() – reads role claim, parses to UserType enum
    Both throw UnauthorizedAccessException if the claim is missing or invalid.

5.8  Password Hashing (Common/Hash/PasswordHasher.cs)
------------------------------------------------------
  PasswordHasher is a static utility using Argon2id from
  Konscious.Security.Cryptography. Parameters:
    SaltSize:             16 bytes (random, generated per password)
    HashSize:             32 bytes
    Iterations:           3
    MemorySize:           131,072 KB = 128 MB
    DegreeOfParallelism:  2

  HashPasswordAsync(password) → Base64(salt[16] || hash[32])
  VerifyPasswordAsync(password, storedBase64) → bool

  Verification uses a constant-time XOR comparison loop (ConstantTimeEquals)
  to prevent timing-based side-channel attacks.

6.3  Entity Configurations  (Persistence/Configurations/)
----------------------------------------------------------
Each entity has a dedicated configuration class implementing
IEntityTypeConfiguration<T>, following the Fluent API pattern.

  UserConfiguration
    – Table: "users"
    – PK: Id (not generated by DB; application assigns Guid)
    – UserType stored as string (max 32)
    – Name max 100, Email max 255, Password max 255
    – Unique index on Email

  LectureConfiguration
    – Table: "lectures"
    – PK: Id (not DB-generated)
    – Status stored as string (max 32)
    – StartTime and Duration required
    – Location mapped via LocationConverter (nullable VARCHAR 256)
    – HasMany(Attendees) with cascade delete; field-backed navigation

  LectureAttendeeConfiguration
    – Table: "lecture_attendees"
    – Composite PK: { LectureId, UserId }
    – FK to Lecture (cascade delete) and to User (restrict)
    – Index on UserId
    – LocationAtJoin mapped via LocationConverter (required)

  UserFollowingConfiguration
    – Table: "user_followings"
    – Composite PK: { FollowedId, FollowerId }
    – FK to User on FollowerId (Restrict) and FollowedId (Restrict)

  QuizConfiguration
    – Table: "quizzes"; HasMany(Questions) with cascade delete; field-backed nav.

  QuizQuestionConfiguration
    – Table: "quiz_questions"; Points with precision (10, 2); HasMany(Options).

  QuizOptionConfiguration
    – Table: "quiz_options"; Text max 512.

  QuizLectureConfiguration
    – Table: "quiz_lectures"; composite index on { LectureId, EndTimeUtc }
      for efficient active-quiz lookup.

  UserAnswerConfiguration
    – Table: "user_answers"
    – Unique index on { UserId, QuizLectureId, QuestionId, OptionId }
      (prevents duplicate answer records)
    – Additional index on { UserId, QuizLectureId } (fast answer retrieval)

  UserSubmissionConfiguration
    – Table: "user_submissions"
    – Score and MaxScore with precision (18, 2)
    – Unique index on { UserId, QuizLectureId } (one submission per student)

6.4  Value Converters  (Persistence/Converters/)
-------------------------------------------------
  LocationConverter  (ValueConverter<Location, string>)
    Converts the Location value object to/from a single database column of type
    VARCHAR(256) with the format "lat,lon,accuracy" in InvariantCulture.
    This avoids the need for a separate GPS table or PostgreSQL geometry types,
    keeping the schema simple while preserving the rich domain value type.

6.5  Generic Repository  (Repositories/GenericRepository.cs)
--------------------------------------------------------------
  An open-generic base class GenericRepository<T> where T : Entity<Guid>
  implements IRepository<T> using EF Core:

    AddAsync    → _dbSet.AddAsync
    Delete      → _dbSet.Remove
    DeleteByIdAsync → FindAsync then Remove (throws KeyNotFoundException if absent)
    GetAllAsync → AsNoTracking().ToListAsync
    GetByIdAsync → FirstOrDefaultAsync on Id
    SaveChangesAsync → _db.SaveChangesAsync
    Update      → _dbSet.Update

  All specialised repositories inherit from this class, overriding or extending
  as needed.

6.6  Specialised Repository Implementations
---------------------------------------------
  All repositories accept AttendanceAppDbContext via primary constructors and
  pass it to GenericRepository<T>.

  LectureRepository
    – GetProfessorLecturesAsync: filters by ProfessorId and optional month/year,
      includes Attendees, orders by StartTime descending, applies Skip/Take.
    – HasProfessorConflictingLectureAsync: loads candidate overlapping lectures
      within a one-day window then performs in-memory interval overlap check.
    – GetStudentLecturesAsync: joins Lecture ↔ LectureAttendee by userId,
      optionally filters by Status and fromDate, ordered by StartTime desc.

  LectureAttendeeRepository
    – GetLectureAttendeesAsync: optional case-insensitive search (ILike from
      Npgsql/EF Functions) across User.Name and User.Email.
    – GetTotalAttendeesAsync: count query with same search predicates.

  QuizRepository
    – GetByIdWithQuestionsAsync: eager loads Questions (ordered by Order).
    – GetByIdWithQuestionsAndOptionsAsync: eager loads Questions and then Options
      via ThenInclude (both ordered).
    – GetQuizzesByTeacherAsync: paged list filtered by ProfessorId and optional
      name substring.

  QuizLectureRepository
    – GetActiveQuizForLectureAsync: finds a QuizLecture whose EndTimeUtc > utcNow.
    – HasActiveQuizForLectureAsync: boolean shortcut using AnyAsync.

  UserAnswerRepository
    – GetByUserQuizLectureQuestionOptionAsync: exact match on four-column
      composite key—used for upsert logic in SaveUserAnswer.

  UserSubmissionRepository
    – GetSubmissionsByQuizLecturePagedAsync: paged results for professors to
      review all student submissions, with optional name/email search.

  UserFollowingsRepository
    – GetFollowingUserIdsAsync / GetFollowerUserIdsAsync: return just Guid lists
      (projection queries) for use by batch-info features.

7.3  Result-to-HTTP Mapping  (Common/ErrorMapper.cs)
-----------------------------------------------------
  The static class ResultActionResultExtensions provides the extension method
  ToActionResult used by all controllers to convert Result/Result<T> to
  IActionResult:

  Success mapping (SuccessKind → HTTP status):
    Ok       → 200 OK (with value body)
    Created  → 201 Created (with value body)
    NoContent→ 204 No Content

  Failure mapping (ErrorKind → HTTP status):
    Validation   → 400 Bad Request
    NotFound     → 404 Not Found
    Conflict     → 409 Conflict
    Unauthorized → 401 Unauthorized
    Forbidden    → 403 Forbidden
    Failure      → 500 Internal Server Error

  When multiple errors are returned, the "most significant" error (determined by
  a priority ranking: Validation > NotFound > Conflict > Unauthorized > Forbidden
  > Failure) is used as the primary HTTP status and title. All errors are
  included in the response body under a ProblemDetails "errors" extension.

CHAPTER 9 – AUTHENTICATION AND SECURITY
================================================================================

9.1  Authentication Flow
-------------------------
1. Client POSTs credentials to POST /api/users/login.
2. LoginUserCommandHandler verifies email and Argon2id password hash.
3. On success, JwtTokenGenerator.GenerateToken produces a signed HS256 JWT
   containing sub (userId), email, role (UserType), and jti (unique token ID).
4. The API controller appends the token as an HttpOnly, Secure, SameSite=Lax
   cookie named "AttendanceApp.Jwt" with a one-hour expiry.
5. Subsequent requests from the browser automatically include this cookie.
6. JwtBearerEvents.OnMessageReceived reads the cookie value and sets
   context.Token so the standard JWT bearer middleware validates it.
7. [Authorize] attributes on controller actions/classes trigger a 401 if the
   token is missing or invalid.

9.2  Authorisation Model
-------------------------
The application uses role-based authorisation via UserType claims. Business-rule
authorisation (e.g. "only the professor who created the lecture may start it")
is enforced within Command handlers, not ASP.NET Core policy middleware.
Handlers verify the authenticated user's ID or type against the resource before
performing mutations.

9.3  Password Security
-----------------------
  – Argon2id with 128 MB memory, 3 iterations, parallelism 2.
  – 16-byte random salt per password; ensures identical passwords hash differently.
  – Constant-time comparison during verification.
  – Password is stored only as a hash; plaintext is never persisted.

9.4  Token Security
-------------------
  – HttpOnly cookie: not accessible from JavaScript; mitigates XSS.
  – Secure flag: transmitted only over HTTPS.
  – SameSite=Lax: allows top-level navigation cross-site but not CSRF-prone
    state-changing requests from third-party sites.
  – JWT has ClockSkew = TimeSpan.Zero for precise expiry enforcement.
  – Secret loaded from environment variable; never hard-coded in source code at
    production level (a development fallback string exists for local dev only).

9.5  Browser Navigation
------------------------
  When an unauthenticated browser request (Accept: text/html) hits a protected
  endpoint, the OnChallenge event redirects to /auth/login instead of returning
  HTTP 401, providing a seamless login experience. AJAX requests still receive
  HTTP 401.

CHAPTER 12 – DATABASE DESIGN
================================================================================

12.1  Entity-Relationship Overview
------------------------------------
The PostgreSQL database contains ten tables. Primary keys are application-
generated GUIDs (UUIDs) to avoid sequential ID leakage and to support
distributed generation.

  users
    id               uuid  PK
    type             varchar(32)  ["Student","Professor","Admin"]
    name             varchar(100) NOT NULL
    email            varchar(255) NOT NULL  UNIQUE
    password         varchar(255) NOT NULL

  lectures
    id               uuid  PK
    professor_id     uuid  NOT NULL  (no FK constraint declared; logical only)
    status           varchar(32)  ["Scheduled","InProgress","Ended","Canceled"]
    name             varchar(?)
    description      varchar(?)
    start_time       timestamp with time zone  NOT NULL
    duration         interval  NOT NULL
    location         varchar(256)  NULLABLE  ["lat,lon,accuracy"]

  lecture_attendees
    lecture_id       uuid  PK (part of composite), FK → lectures ON DELETE CASCADE
    user_id          uuid  PK (part of composite), FK → users ON DELETE RESTRICT
    time_joined      timestamp with time zone
    location_at_join varchar(256)  NOT NULL
    INDEX (user_id)

  user_followings
    followed_id      uuid  PK (part of composite), FK → users ON DELETE RESTRICT
    follower_id      uuid  PK (part of composite), FK → users ON DELETE RESTRICT

  quizzes
    id               uuid  PK
    name             varchar(256)  NOT NULL
    duration         interval  NOT NULL
    professor_id     uuid  NOT NULL
    created_at_utc   timestamp with time zone  NOT NULL

  quiz_questions
    id               uuid  PK
    quiz_id          uuid  NOT NULL  FK → quizzes ON DELETE CASCADE
    text             varchar(1024)  NOT NULL
    order            int  NOT NULL
    points           numeric(10,2)  NULLABLE

  quiz_options
    id               uuid  PK
    question_id      uuid  NOT NULL  FK → quiz_questions ON DELETE CASCADE
    text             varchar(512)  NOT NULL
    order            int  NOT NULL
    is_correct       bool  NOT NULL

  quiz_lectures
    id               uuid  PK
    lecture_id       uuid  NOT NULL
    quiz_id          uuid  NOT NULL
    created_at_utc   timestamp with time zone  NOT NULL
    end_time_utc     timestamp with time zone  NOT NULL
    INDEX (lecture_id, end_time_utc)

  user_answers
    id               uuid  PK
    user_id          uuid  NOT NULL
    quiz_lecture_id  uuid  NOT NULL
    question_id      uuid  NOT NULL
    option_id        uuid  NOT NULL
    choice           bool  NOT NULL
    UNIQUE (user_id, quiz_lecture_id, question_id, option_id)
    INDEX (user_id, quiz_lecture_id)

  user_submissions
    id               uuid  PK
    user_id          uuid  NOT NULL
    quiz_lecture_id  uuid  NOT NULL
    submitted        bool  NOT NULL
    submitted_at_utc timestamp with time zone  NOT NULL
    score            numeric(18,2)  NOT NULL
    max_score        numeric(18,2)  NOT NULL
    UNIQUE (user_id, quiz_lecture_id)

12.2  Key Design Decisions
---------------------------
  – Application-generated GUIDs (ValueGeneratedNever) prevent the DB from
    assigning sequential integer IDs, which could leak row count information.
  – Composite PKs on lecture_attendees and user_followings serve as natural
    unique constraints, preventing duplicates without extra indexes.
  – Status and type enums stored as strings rather than integers for
    readability in SQL queries and migration readability.
  – Location stored as a single VARCHAR column rather than PostGIS geometry,
    keeping dependencies minimal while still enabling Haversine computation
    at the application level.
  – Cascade delete from lectures → lecture_attendees (removing a lecture
    removes all attendance records). No cascade on users; attendee records
    are preserved if a user account is deleted (restrict).
  – The quiz_lectures composite index on (lecture_id, end_time_utc) supports
    the "get active quiz for lecture" query which filters by both columns.
  – user_answers unique index prevents a student from recording two different
    answers for the same option in the same quiz-lecture (toggle goes through
    an upsert-like UpdateChoice flow).
  – user_submissions unique index (user_id, quiz_lecture_id) enforces one
    submission per student per quiz-lecture at the database level.

CHAPTER 13 – DESIGN PATTERNS AND PRINCIPLES APPLIED
================================================================================

13.1  Clean Architecture / Hexagonal Architecture
--------------------------------------------------
The application rigidly follows the Dependency Rule: inner layers have no
reference to outer layers. The Domain has zero external dependencies. The
Application depends only on the Domain. Infrastructure and presentation layers
depend on the inner layers but not on each other (except through the composition
root in Web).

13.2  Domain-Driven Design (DDD)
---------------------------------
Key DDD concepts applied:

  Entities – objects with persistent identity (User, Lecture, LectureAttendee,
    Quiz, QuizQuestion, QuizOption, QuizLecture, UserAnswer, UserSubmission,
    UserFollowing).

  Aggregate Roots – the consistency boundary owners: User, Lecture, Quiz.
    External code may only modify aggregates through their public methods; inner
    collections are hidden behind read-only properties.

  Value Objects – Location is a readonly record struct implementing equality by
    structural value rather than reference identity. It has no identity (no Id),
    is immutable, and encapsulates the Haversine logic.

  Domain Exceptions – DomainException is the domain's exclusive error signal;
    all business-rule violations throw this type.

  Repository Interfaces – defined in the Domain (ILectureRepository etc.) with
    implementations in Infrastructure, decoupling domain logic from persistence.

  Guard Clauses – internal Guard class enforces invariants at the earliest
    possible moment (object construction), making invalid states impossible.

  Factory Methods – Quiz.Create and QuizLecture.Create are static factory
    methods that validate inputs before constructing the object, preventing
    partially initialised aggregates from existing.

13.3  Command Query Responsibility Segregation (CQRS)
-----------------------------------------------------
Commands (mutations) and Queries (reads) are strictly separated into distinct
MediatR request/handler pairs. This provides clarity of intent, independent
testability, and the option to scale read and write paths independently in the
future.

13.4  Repository Pattern
-------------------------
All data access is abstracted behind repository interfaces. Controllers and
handlers depend on interfaces (ILectureRepository, IUserRepository, etc.), never
on DbContext directly. This enables:
  – Easy substitution in tests (InMemory EF replaces Npgsql).
  – Single responsibility: repositories encapsulate all query logic.
  – TDD-friendly: handlers can be unit-tested against mock repositories.

13.5  Pipeline / Decorator Pattern (MediatR Behaviors)
------------------------------------------------------
Cross-cutting concerns (validation, exception-to-result mapping) are applied to
all handlers uniformly via the MediatR pipeline. This keeps individual handlers
clean from try/catch boilerplate and FluentValidation invocation.
The ordering of behaviours matters:
  1. ExceptionToResultBehavior (outermost) catches exceptions thrown by inner layers
  2. ValidationBehavior validates the request BEFORE the handler runs

13.6  Railway-Oriented Programming (Result Type)
-------------------------------------------------
The Result<T> / Result types implement a lightweight version of ROP, a functional
programming pattern popularised by Scott Wlaschin. Instead of exceptions for
control flow, operations return a discriminated union of success or failure.
Callers use Match, Map, Bind, Tap to chain operations without defensive null
checks. The ToActionResult extension method is the final interpreter that
translates Result semantics into HTTP semantics.

13.7  Value Converter Pattern (EF Core)
-----------------------------------------
The LocationConverter demonstrates the EF Core Value Converter pattern to
serialise a rich domain value type (Location struct) to a simple database column
type (VARCHAR). This keeps the domain free of persistence concerns while still
storing the data efficiently.

13.8  Extension Method Pattern
--------------------------------
Extension methods are used extensively:
  – ClaimsPrincipalExtensions.GetUserId() / GetUserType() for clean claim extraction.
  – ResultActionResultExtensions.ToActionResult() for HTTP mapping.
  – HttpContextExtensions.ReadAsAsync<T>() in test helpers.

13.9  Options Pattern / Configuration
---------------------------------------
Connection strings and JWT secrets are externalised via environment variables
loaded by DotNetEnv. This follows the Twelve-Factor App methodology, keeping
sensitive configuration out of source control.

13.10  Geofencing with Haversine Distance
------------------------------------------
The attendance join feature implements a 100-metre geofence using the Haversine
formula entirely within the Domain layer's Location value type. This is a clean
example of domain logic that does not bleed into infrastructure or application
layers—the distance computation is a pure function on two location values.

14.1  Summary
--------------
AttendanceApp is a comprehensive full-stack web application implementing a
GPS-based lecture attendance and in-lecture quiz platform. The project
demonstrates the application of mature software engineering practices:

  – Clean Architecture with strict layer separation.
  – Domain-Driven Design with rich aggregate roots, value objects, and guard
    clauses that make invalid domain states impossible to represent.
  – CQRS via MediatR, organising 40+ use-cases into discrete, testable units.
  – Railway-Oriented Programming via a custom Result<T> monad.
  – MediatR pipeline behaviors for cross-cutting validation and error handling.
  – Argon2id password hashing meeting OWASP standards.
  – HTTP-only JWT cookie authentication resistant to XSS.
  – Geofenced attendance using the Haversine formula implemented in the domain.
  – EF Core Code-First with Fluent API configurations, value converters, and a
    clean migration history.
  – A server-rendered MVC front end with per-view JavaScript modules.
  – Comprehensive test coverage: 10 domain entity test classes, 1 result type
    test class, 1 API mapping test class, and 9 integration test controller
    classes covering all major HTTP endpoint behaviours.
  – Containerised infrastructure using Docker Compose for one-command startup.

14.2  Potential Future Enhancements
-------------------------------------
  1. Real-Time Notifications via WebSockets or SignalR
       Rather than polling, a real-time hub could push lecture status changes
       (Started, Ended) and quiz activations directly to connected students.

  2. OAuth2 / OpenID Connect Integration
       Social login (Google, Microsoft) for students and SSO with university
       identity providers via OIDC.

  3. Refresh Tokens
       The current JWT implementation uses a single one-hour token. A refresh
       token scheme would allow sessions to persist without re-login.

  4. Fine-Grained Role-Based Access Control
       A policy-based authorisation system could replace the in-handler checks,
       enabling declarative attribute-level security.

  5. Admin Dashboard
       The UserType.Admin role exists in the domain but no admin-specific UI or
       endpoints have been implemented.

  6. Media Uploads
       Allowing professors to attach materials (slides, PDFs) to lectures,
       stored in cloud blob storage with URLs persisted in the lecture record.

  7. Analytics and Reporting
       Aggregate attendance statistics per professor / course / date range
       exported as CSV or visualised in-app.

  8. Mobile Application
       A native iOS/Android app using the GPS sensor natively (better accuracy
       than browser Geolocation API) for attendance and quiz participation.

  9. PostGIS Geospatial Extension
       For more complex proximity queries (e.g. campus-boundary checks),
       replacing the VARCHAR location column with PostGIS geometry types would
       enable server-side spatial queries.

  10. Event Sourcing for Attendance Audit Trail
       Recording every attendance event (join, leave, re-join) with timestamps
       to provide a full audit log for dispute resolution.
