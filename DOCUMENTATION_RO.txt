CAPITOLUL 1 – PREZENTAREA PROIECTULUI ȘI MOTIVAȚIE
================================================================================

1.1  Introducere
-----------------
AttendanceApp este un sistem web de management al prezenței academice, conceput
pentru a susține mediile universitare moderne în care studenții și profesorii
interacționează digital în jurul cursurilor, quiz-urilor și participării la curs.
Sistemul permite profesorilor să programeze și să gestioneze cursuri, să pornească
sesiuni în timp real în clasă, să solicite studenților autentificarea folosind
geofencing bazat pe GPS și să lanseze quiz-uri live în timpul acelor sesiuni.
Studenții, la rândul lor, urmăresc profesorii, se alătură cursurilor în desfășurare
prin trimiterea locației GPS, răspund la întrebări de quiz în timp real și
trimit răspunsurile finale la quiz.

Aplicația a fost construită ca un sistem full-stack care cuprinde un back-end
RESTful Web API, un front-end MVC randat pe server, o bază de date relațională
PostgreSQL și o suită cuprinzătoare de teste unitare și de integrare. Soluția
este containerizată folosind Docker astfel încât baza de date și serviciile
auxiliare să poată fi pornite cu o singură comandă.

1.2  Enunțul problemei
-----------------------
Gestionarea tradițională a prezenței în universități se bazează pe registre pe
hârtie sau pe simple foi de semnătură, care sunt susceptibile la fraudă
(studenți care semnează în locul colegilor absenți), costuri administrative
ridicate și lipsa vizibilității în timp real pentru profesori. Soluțiile
digitale care necesită hardware dedicat (badge-uri NFC, scanere biometrice)
implică costuri mari de infrastructură.

AttendanceApp abordează această problemă prin:
  – Utilizarea senzorului GPS deja prezent în fiecare smartphone pentru a verifica
    proximitatea fizică a unui student față de sala de curs (în limita a 100 de metri).
  – Oferirea către profesori a unui tablou de bord web instant care arată cine este prezent.
  – Permiterea evaluărilor de tip quiz în timpul cursului, ale căror rezultate și
    punctaje sunt calculate automat și disponibile imediat după trimitere.
  – Încorporarea unui mecanism social de follow, astfel încât studenții să poată
    descoperi și urmări profesorii ale căror cursuri îi interesează.

1.3  Domeniul de aplicare al sistemului
----------------------------------------
Sistemul gestionează următoarele cazuri de utilizare principale:

  1. Înregistrare și autentificare utilizatori (studenți, profesori, administratori).
  2. Gestionarea ciclului de viață al cursurilor: programare, pornire (cu GPS),
     încheiere, anulare și ștergere.
  3. Urmărirea prezenței: studenții se alătură cursurilor în desfășurare furnizând
     coordonatele GPS curente; sistemul respinge o cerere de alăturare dacă
     distanța Haversine calculată până la locația cursului depășește 100 de metri.
  4. Gestionarea quiz-urilor: profesorii creează quiz-uri, adaugă întrebări și
     opțiuni cu alegere multiplă și activează un quiz pentru un anumit curs live.
  5. Participarea la quiz: studenții văd quiz-ul activ, răspund la întrebări și
     trimit quiz-ul completat; sistemul calculează și stochează punctajul.
  6. Follow social: studenții urmăresc profesori; numărul de urmăritori/urmăriți
     este vizibil în paginile de profil.
  7. Interogări paginate cu filtrare opțională pentru toate entitățile majore.

CAPITOLUL 2 – PROIECTARE ARHITECTURALĂ
================================================================================

2.1  Clean Architecture
------------------------
AttendanceApp urmează modelul Clean Architecture (cunoscut și ca Onion
Architecture sau Hexagonal Architecture), formulat de Robert C. Martin.
Principiul fundamental al Clean Architecture este Dependency Rule: dependențele
codului sursă indică întotdeauna spre interior—către politici de nivel superior—
și niciodată spre exterior, către detalii de implementare.

Soluția este împărțită în cinci proiecte deployabile / compilabile plus două
proiecte de test, organizate în următoarele straturi concentrice (de la interior la exterior):

  +----------------------------------------------------+
  |           4. AttendanceApp.Web  (MVC UI)           |
  |     3. AttendanceApp.Api  (REST Controllers)       |
  |   2. AttendanceApp.Infrastructure  (EF Core, DB)   |
  |   1. AttendanceApp.Application  (Use Cases, CQRS)  |
  |       0. AttendanceApp.Domain  (Entities, Rules)   |
  +----------------------------------------------------+

    AttendanceApp.UnitTests         (testează Domain + Application + Api)
    AttendanceApp.IntegrationTests  (testează Web + Api prin EF in-memory)

Săgeți de dependență:

  Domain        ←  Application
  Application   ←  Infrastructure
  Application   ←  Api
  Domain        ←  Infrastructure
  Domain + Infrastructure + Application + Api  ←  Web

Proiectul Domain NU are nicio dependență externă NuGet. Proiectul Application
depinde doar de Domain plus trei pachete NuGet (MediatR,
FluentValidation, Konscious Argon2). Infrastructure este singurul proiect care
cunoaște Entity Framework Core și PostgreSQL. Proiectul API cunoaște
controller-ele ASP.NET Core și Swashbuckle, dar nu referențiază Infrastructure
direct—accesul se face prin interfețele Application.

2.2  Graficul dependențelor între proiecte
-------------------------------------------
Soluția Visual Studio (AttendanceApp.sln) înregistrează șapte proiecte:

  AttendanceApp.Domain          – fără referințe la alte proiecte
  AttendanceApp.Application     – → Domain
  AttendanceApp.Infrastructure  – → Application   (și tranzitiv → Domain)
  AttendanceApp.Api             – → Infrastructure (și tranzitiv → toate)
  AttendanceApp.Web             – → Domain, Application, Infrastructure, Api
  AttendanceApp.UnitTests       – → Domain, Application, Api
  AttendanceApp.IntegrationTests– → Api, Web, Infrastructure

2.3  Responsabilitățile straturilor
------------------------------------

  Stratul Domain
    Definește entitățile de business de bază, obiectele de valoare, excepțiile
    de domeniu și interfețele repository. Nu conține cunoaștere despre framework
    sau baza de date.

  Stratul Application
    Codifică use-case-urile aplicației utilizând modelul CQRS prin MediatR.
    Fiecare use case este implementat ca un Command (mutație) sau Query (citire),
    asociat cu un handler și un validator FluentValidation. Aspectele transversale
    (pipeline de validare, conversie excepții) sunt tratate prin comportamente
    de pipeline MediatR.

  Stratul Infrastructure
    Implementează interfețele repository declarate în Domain folosind Entity
    Framework Core 10 și provider-ul Npgsql pentru PostgreSQL. Conține
    DbContext-ul, toate configurațiile de entități Fluent API, convertoare
    de valori personalizate și istoricul migrațiilor EF.

  Stratul API (AttendanceApp.Api)
    Un proiect ASP.NET Core minimal care înregistrează Swagger/OpenAPI.
    Controller-ele sale folosesc IMediator injectat prin constructor pentru a
    trimite comenzi/interogări și convertesc valorile Result<T> în răspunsuri HTTP
    printr-o metodă de extensie partajată.

  Stratul Web (AttendanceApp.Web)
    Un proiect ASP.NET Core MVC care servește view-uri Razor randate pe server
    ca front-end orientat către utilizator. Găzduiește toate înregistrările de
    servicii (configurarea containerului DI, autentificare JWT bearer, EF,
    MediatR, repository-uri) și folosește aceleași controller-e API prin composition root.
    Resursele statice front-end (Bootstrap 5, jQuery, CSS și JavaScript personalizat)
    sunt servite din wwwroot.

2.4  Fluxul unei cereri
------------------------
O cerere HTTP tipică urmează acest pipeline:

  Browser / API Client
      |
      v
  ASP.NET Core Routing  (pipeline middleware în Program.cs)
      |
      v
  Authentication Middleware  (validare JWT cookie/header)
      |
      v
  Authorization Middleware   (verificare atribut [Authorize])
      |
      v
  Controller Action          (ex. LectureController.CreateLecture)
      |  extrage ID-ul utilizatorului autentificat din ClaimsPrincipal
      |  mapează request DTO → comandă MediatR
      v
  IMediator.Send(command)
      |
      v
  MediatR Pipeline
      |  1. ExceptionToResultBehavior  (prinde excepții → Result.Fail)
      |  2. ValidationBehavior         (rulează FluentValidation)
      v
  Command / Query Handler   (logică de business, apelează interfețe repository)
      |
      v
  EF Core Repository        (interogare SQL generată → Npgsql → PostgreSQL)
      |
      v
  Result<T>  returnat înapoi pe call stack
      |
      v
  Extensie ToActionResult()  (mapează SuccessKind / ErrorKind → cod status HTTP)
      |
      v
  HTTP Response (corp JSON sau ProblemDetails)

    MediatR este mecanismul central de dispatch pentru comenzi și interogări.

    Application pentru a localiza automat toate handler-ele la startup:
      builder.Services.AddMediatR(cfg =>

CAPITOLUL 4 – STRATUL DOMAIN (AttendanceApp.Domain)
================================================================================

4.1  Prezentare generală
-------------------------
Stratul Domain este stratul cel mai interior al arhitecturii. Este o bibliotecă
de clase pură (Microsoft.NET.Sdk, țintă net10.0) fără pachete NuGet externe.
Definește modelul de business al sistemului: entități, aggregate roots, obiecte
de valoare, enumerări, excepții de domeniu, clauze guard și interfețe repository.

4.2  Blocuri constructive comune
--------------------------------

  4.2.1  Entity<TId>  (Common/Entity.cs)
  ----------------------------------------
  Clasa de bază generică pentru toate obiectele de domeniu care au o identitate persistentă.

    public abstract class Entity<TId> where TId : notnull
    {
        public TId Id { get; protected set; } = default!;
        protected Entity(TId id) => Id = id;
        protected Entity() { }

        public override bool Equals(object? obj)
            => obj is Entity<TId> other
               && EqualityComparer<TId>.Default.Equals(Id, other.Id);

        public override int GetHashCode() => Id.GetHashCode();
    }

  Egalitatea de identitate este impusă: două instanțe Entity sunt egale dacă și
  numai dacă Id-urile lor sunt egale. Aceasta implementează conceptul DDD
  (Domain-Driven Design) al identității entității. Constructorul protected fără
  argumente există exclusiv pentru a satisface cerința EF Core pentru un
  constructor fără parametri în timpul materializării obiectelor.

  4.2.2  AggregateRoot<TId>  (Common/AggregateRoot.cs)
  -------------------------------------------------------
  Subclasă subțire a Entity<TId> care marchează o entitate ca aggregate root—
  singurul punct de intrare prin care sunt aplicate toate invariantele agregatului.

    public abstract class AggregateRoot<TId> : Entity<TId> where TId : notnull

  În DDD, aggregate roots sunt singurele obiecte stocate de repository-uri.
  Sistemul are trei aggregate roots: User, Lecture și Quiz.

  4.2.3  DomainException  (Common/DomainException.cs)
  -----------------------------------------------------
  Un tip de excepție sealed aruncat exclusiv de entitățile de domeniu și Guard
  pentru a semnala încălcări ale regulilor de business. Extinde Exception și
  primește un mesaj.

    public sealed class DomainException(string message) : Exception(message)

  Acest tip este capturat de ExceptionToResultBehavior în stratul Application și
  convertit într-un Error de tip ErrorKind.Validation, producând un răspuns HTTP
  400 Bad Request fără a expune stack trace-uri.

  4.2.4  Guard  (Common/Guard.cs)
  ---------------------------------
  O clasă utilitară statică internă ce oferă verificări de precondiții folosite
  în constructorii și metodele entităților. Metodele aruncă DomainException la încălcare:

    Guard.NotNull(value, name)
      – Aruncă dacă value este null.
    Guard.NotNullOrWhiteSpace(value, name, maxLen?)
      – Aruncă dacă șirul este null, gol, doar spații sau depășește maxLen opțional.
    Guard.InRange(value, name, min, max)
      – Aruncă dacă întregul este în afara intervalului [min, max].
    Guard.Positive(decimal value, name)
      – Aruncă dacă value <= 0.
    Guard.Positive(TimeSpan value, name)
      – Aruncă dacă TimeSpan <= TimeSpan.Zero.
    Guard.NotNegative(decimal value, name)
      – Aruncă dacă value < 0.
    Guard.NotEmpty(Guid value, name)
      – Aruncă dacă Guid == Guid.Empty.
    Guard.NotInPast(DateTime value, name)
      – Aruncă dacă DateTime < DateTime.UtcNow.

  Clasa este marcată internal astfel încât să nu poată fi utilizată în afara
  assembly-ului Domain, menținând aplicarea invariantelor de domeniu în interiorul domeniului.

  4.2.5  Location  (Common/Locartion.cs)
  ----------------------------------------
  (Notă: fișierul se numește Locartion.cs cu o eroare de scriere; tipul este
  denumit corect Location.)

  Un readonly record struct care reprezintă un triplet de coordonate GPS:

    public readonly record struct Location
    {
        public double Latitude  { get; init; }
        public double Longitude { get; init; }
        public double Accuracy  { get; init; }
    }

  Fiind tip valoare (struct) și imuabil (readonly record), două Location cu
  aceeași Latitude, Longitude și Accuracy sunt considerate aceeași valoare.

  Metoda DistanceTo(Location other) implementează formula Haversine pentru a
  calcula distanța pe cercul mare, în metri, între două coordonate geografice:

    d = 2R * arcsin( sqrt(
          sin²(Δlat/2) + cos(lat1)*cos(lat2)*sin²(Δlon/2)
        ))

  unde R = 6,371,000 metri (raza medie a Pământului).

  ToString() serializează în "lat,lon,accuracy" (separat prin virgulă, cultură
  invariantă), iar static FromString(string) parsează această reprezentare înapoi.
  Această formă string este folosită de LocationConverter din stratul Infrastructure
  pentru a stoca valorile Location în baza de date ca o singură coloană VARCHAR.

4.3  Enumerări
---------------

  4.3.1  LectureStatus  (Enums/LectureStatus.cs)
  ------------------------------------------------
  Reprezintă starea ciclului de viață al unei entități Lecture:

    Scheduled = 0    – Starea implicită când un curs este creat.
    InProgress = 1   – Profesorul a apelat Start(), furnizând o locație.
    Ended = 2        – Profesorul a apelat End().
    Canceled = 3     – Cursul a fost anulat prin ChangeStatus.

  Tranzițiile de stare sunt impuse de metodele de domeniu din Lecture:
    – Start() necesită Status == Scheduled; altfel aruncă DomainException.
    – End() necesită Status == InProgress; altfel aruncă DomainException.
    – ChangeStatus() permite orice tranziție fără guard (folosit pentru admin/anulare).

  4.3.2  UserType  (Enums/UserType.cs)
  --------------------------------------
  Clasificarea rolurilor pentru utilizatorii sistemului:

    Student   = 0
    Professor = 1
    Admin     = 2

  Regulile de business depind de UserType. De exemplu, doar profesorii (și adminii)
  pot crea cursuri; doar studenții se pot alătura cursurilor.

4.4  Agregat: Lecture  (Lectures/)
-----------------------------------

  4.4.1  Lecture  (Lectures/Lecture.cs)
  ----------------------------------------
  Aggregate root care reprezintă o sesiune academică programată.

  Proprietăți:
    Guid         Id            – setat în constructor, niciodată regenerat
    Guid         ProfessorId   – referință imuabilă către creator
    LectureStatus Status       – mutabil, tranziții prin Start/End/ChangeStatus
    string       Name          – până la 256 de caractere
    string       Description   – text liber
    DateTime     StartTime     – trebuie să fie în viitor la momentul construirii
    TimeSpan     Duration      – trebuie să fie pozitivă
    Location?    Location      – null până la pornirea cursului; setată de Start()
    DateTime     EndTime       – calculat: StartTime + Duration (nu este mapat în DB)
    IReadOnlyCollection<LectureAttendee> Attendees – proiecție read-only a
                                private List<LectureAttendee> _attendees

  Constructorul validează toate câmpurile prin Guard și setează starea inițială
  Scheduled fără Location.

  Operații de domeniu:
    void Start(Location location)
      – Validează că location nu este null.
      – Necesită Status == Scheduled; altfel aruncă DomainException.
      – Tranziționează Status → InProgress și înregistrează Location.

    void End()
      – Necesită Status == InProgress; altfel aruncă DomainException.
      – Tranziționează Status → Ended.

    void ChangeStatus(LectureStatus status)
      – Atribuire de stare fără verificări (folosită pentru suprascrieri administrative).

  Notă ORM: Proprietatea EndTime este marcată [NotMapped] deoarece este calculată
  și nu trebuie persisată. Câmpul _attendees reprezintă stocarea efectivă;
  proprietatea publică Attendees expune un wrapper read-only.

  4.4.2  LectureAttendee  (Lectures/LectureAttendee.cs)
  -------------------------------------------------------
  O entitate (nu un aggregate root) care reprezintă prezența înregistrată a
  unui student la un anumit curs.

  Proprietăți:
    Guid     Id              – generat prin Guid.NewGuid() în constructor
    Guid     LectureId       – FK către Lecture
    Guid     UserId          – FK către User
    DateTime TimeJoined      – setat la DateTime.UtcNow în constructor
    Location LocationAtJoin  – coordonate GPS capturate la momentul alăturării

  Constructorul validează că LectureId și UserId nu sunt goale prin Guard.

  Rețineți că LectureAttendee este stocat ca element copil al agregatului Lecture,
  dar este persisat într-un tabel separat. EF Core urmărește proprietatea de
  navigare Attendees din Lecture pentru a le persista/încărca.

4.5  Agregat: User  (Users/)
-----------------------------

  4.5.1  User  (Users/User.cs)
  ------------------------------
  Aggregate root care reprezintă un participant înregistrat în sistem.

  Proprietăți:
    Guid     Id       – setat în constructor
    UserType Type     – mutabil (poate fi schimbat de admin)
    string   Name     – trunchiat; max 100 caractere
    string   Email    – trunchiat; max 255 caractere; unic în baza de date
    string   Password – stochează hash-ul Argon2id, nu textul în clar

  Operații de domeniu:
    void ChangeName(string name)
    void ChangeEmail(string email)
    void ChangePassword(string password)
    void ChangeType(UserType type)

  Toți setter-ii validează prin Guard. Câmpul parolă conține hash-ul Argon2id
  calculat în stratul Application înainte ca User să fie construit.

  Notă: Entitatea are și un private List<Guid> _attendedLectures și corespondentul
  său read-only (IReadOnlyCollection<Guid> AttendedLectures). Acesta este un
  artefact de design rămas; relația reală de prezență este gestionată prin
  entitatea LectureAttendee și repository-ul ei.

  4.5.2  UserFollowing  (Users/UserFollowing.cs)
  ------------------------------------------------
  Entitate care reprezintă o relație direcționată de follow social între doi utilizatori.

  Proprietăți:
    Guid     Id         – atribuit prin Guid.NewGuid()
    Guid     FollowerId – utilizatorul care urmărește
    Guid     FollowedId – utilizatorul urmărit
    DateTime FollowedAt – timestamp-ul evenimentului de follow

  Constructorul impune:
    – Niciun ID nu este gol (Guard.NotEmpty).
    – FollowerId != FollowedId (self-follow aruncă DomainException).

4.6  Agregat: Quiz  (Quizzes/)
--------------------------------

  4.6.1  Quiz  (Quizzes/Quiz.cs)
  --------------------------------
  Aggregate root care reprezintă un set reutilizabil de întrebări creat de un profesor.

  Proprietăți:
    Guid     Id           – atribuit în constructorul privat
    string   Name         – trunchiat; obligatoriu
    TimeSpan Duration     – trebuie să fie pozitivă (definește cât timp au studenții)
    Guid     ProfessorId  – referință către profesorul creator
    DateTime CreatedAtUtc – timestamp de creare
    IReadOnlyCollection<QuizQuestion> Questions

  Constructorul este privat; crearea se face prin metoda factory statică
    Quiz.Create(string name, TimeSpan duration, Guid professorId, DateTime now)
  care validează intrările cu Guard înainte de a apela constructorul. Acest model
  factory asigură că agregatul este întotdeauna valid la creare.

  Operații de domeniu:
    void Rename(string newName)
    void ChangeDuration(TimeSpan newDuration)
    QuizQuestion AddQuestion(string text, int order, decimal? points)
    void RemoveQuestion(Guid questionId)

  4.6.2  QuizQuestion  (Quizzes/QuizQuestion.cs)
  -------------------------------------------------
  Entitate care reprezintă o singură întrebare dintr-un quiz.

  Proprietăți:
    Guid     Id      – atribuit de factory-ul static Create
    Guid     QuizId  – FK către Quiz-ul părinte
    string   Text    – trunchiat; max 1024 caractere
    int      Order   – ordine de afișare; trebuie să fie >= 1
    decimal? Points  – nullable; dacă este setat, trebuie să fie nenegativ
    IReadOnlyCollection<QuizOption> Options

  Creată exclusiv prin Quiz.AddQuestion (care apelează intern QuizQuestion.Create).
  Operațiile de domeniu includ ChangeText, ChangeOrder, ChangePoints,
  AddOption, RemoveOption și ClearCorrectOptions.

  4.6.3  QuizOption  (Quizzes/QuizOption.cs)
  --------------------------------------------
  Entitate care reprezintă o singură opțiune cu alegere multiplă pentru o întrebare.

  Proprietăți:
    Guid   Id         – atribuit prin Create
    Guid   QuestionId – FK către QuizQuestion părinte
    string Text       – trunchiat; max 512 caractere
    int    Order      – ordine de afișare; trebuie să fie >= 1
    bool   IsCorrect  – dacă această opțiune este un răspuns corect

  Creată prin QuizQuestion.AddOption. Operații de domeniu:
    void ChangeText(string), void ChangeOrder(int),
    void MarkAsCorrect(), void MarkAsIncorrect().

  4.6.4  QuizLecture  (Quizzes/QuizLecture.cs)
  ----------------------------------------------
  Entitate care reprezintă activarea unui Quiz pentru o sesiune Lecture live
  specifică. Capturează momentul de start (CreatedAtUtc) și timpul de încheiere
  calculat (EndTimeUtc = CreatedAtUtc + quiz.Duration).

  Metodă cheie:
    bool IsActive(DateTime utcNow) => utcNow < EndTimeUtc;
  Folosită de repository pentru a determina dacă studenții mai pot răspunde la quiz.

  Creată prin factory-ul static QuizLecture.Create.

  4.6.5  UserAnswer  (Quizzes/UserAnswer.cs)
  --------------------------------------------
  Entitate care înregistrează selecția unui student (alegere true/false) pentru o
  opțiune specifică într-o instanță specifică de quiz-lecture.

  Proprietăți:
    Guid UserId, QuizLectureId, QuestionId, OptionId, bool Choice.

  Studenții își pot modifica alegerea înainte de trimiterea finală; UpdateChoice(bool)
  actualizează câmpul Choice.

  4.6.6  UserSubmission  (Quizzes/UserSubmission.cs)
  ----------------------------------------------------
  Entitate care reprezintă trimiterea finală, punctată, a unui student pentru un quiz-lecture.

  Proprietăți:
    Guid    UserId, QuizLectureId
    bool    Submitted        – întotdeauna true la creare
    DateTime SubmittedAtUtc
    decimal Score            – punctaj calculat
    decimal MaxScore         – punctaj maxim posibil

  Creată prin factory-ul static UserSubmission.Create. Odată creată, este
  imuabilă (fără metode de mutație).

CAPITOLUL 5 – STRATUL APPLICATION (AttendanceApp.Application)
================================================================================

5.1  Prezentare generală
-------------------------
Stratul Application codifică use-case-urile la nivel de aplicație folosind
modelul CQRS (Command Query Responsibility Segregation) implementat prin MediatR.
Referențiază doar stratul Domain și propriile pachete NuGet—nu are cunoaștere
despre EF Core, HTTP sau baza de date. Acest lucru asigură testabilitatea completă
a logicii de business în izolare.

Assembly-ul exportă o singură interfață publică IAssemblyMarker (în IAssemblyMaker.cs
— observați mica eroare de scriere din numele fișierului), al cărei unic scop
este să servească drept ancoră de tip pentru scanarea assembly-ului de către
MediatR și FluentValidation:

  public interface IAssemblyMarker { }

5.2  CQRS cu MediatR
---------------------
Fiecare caz de utilizare este exprimat fie ca Command (mutație cu efecte secundare),
fie ca Query (preluare doar în citire):

  Comenzile implementează IRequest<TResponse>:
    – Tipurile de returnare sunt de regulă Result<Guid> (pentru operații de creare)
      sau Result / Result<T> (pentru alte mutații).
    – Exemple: CreateUserCommand, CreateLectureCommand, JoinLectureCommand,
      SubmitQuizCommand, SaveUserAnswerCommand, ToggleFollowUserCommand.

  Interogările implementează IRequest<TResponse>:
    – Tipurile de returnare sunt de regulă Result<SomeDto> sau Result<PagedDto>.
    – Exemple: GetProfessorLecturesQuery, GetStudentLecturesQuery,
      GetQuizzesByProfessorQuery, GetActiveQuizForLectureQuery,
      SearchUsersByNameQuery.

  Convenție de denumire pentru handler:
    Sufix CommandHandler sau QueryHandler pe clasa de implementare. De
    exemplu, CreateUserCommandHandler : IRequestHandler<CreateUserCommand, Result<Guid>>.

5.4  Prezentări detaliate ale unor funcționalități selectate
-------------------------------------------------------------

  5.4.1  CreateUser (Înregistrare)
  ----------------------------------
  Comandă:  CreateUserCommand(Name, Email, Password, UserType)
  Handler:  CreateUserCommandHandler
    1. Verifică unicitatea email-ului prin IUserRepository.GetByEmailAsync.
    2. Apelează PasswordHasher.HashPasswordAsync (Argon2id) pentru a produce hash-ul.
    3. Construiește un User nou (Guid.NewGuid(), type, name, email, hash).
    4. Persistă prin IUserRepository.AddAsync + SaveChangesAsync.
    5. Returnează Result<Guid>.Created(newUserId).

  Validator (CreateUserCommandValidator):
    – Name: obligatoriu, max 100 caractere.
    – Email: obligatoriu, max 255 caractere, trebuie format valid de email.
    – Password: obligatoriu, min 8 caractere, max 255 caractere.
    – UserType: trebuie să fie o valoare validă de enum.

  5.4.2  LoginUser (Autentificare)
  ----------------------------------
  Comandă:  LoginUserCommand(Email, Password)
  Handler:  LoginUserCommandHandler
    1. Preia utilizatorul după email; aruncă KeyNotFoundException dacă nu există.
    2. Verifică parola prin PasswordHasher.VerifyPasswordAsync.
    3. La succes, apelează JwtTokenGenerator.GenerateToken pentru a produce un JWT.
    4. Returnează Result<string>.Ok(jwt).
  Șirul JWT este returnat către controller, care îl atașează ca cookie HTTP-only
  (SameSite=Lax, Secure, MaxAge=1 oră).

  5.4.3  CreateLecture
  ---------------------
  Comandă:  CreateLectureCommand(ProfessorId, Name, Description, StartTime, Duration)
  Handler:  CreateLectureCommandHandler
    1. Verifică existența utilizatorului profesor.
    2. Verifică UserType != Student (doar profesorii/adminii pot crea).
    3. Detectează conflicte de programare prin
       ILectureRepository.HasProfessorConflictingLectureAsync.
    4. Construiește agregatul Lecture, persistă, returnează Result<Guid>.Created.

  5.4.4  JoinLecture (Prezență geofencing)
  -----------------------------------------
  Comandă:  JoinLectureCommand(UserId, LectureId, Position)
    – Position este un string "lat,lon,accuracy" pars-at într-un obiect de valoare Location.
  Handler:  JoinLectureCommandHandler
    1. Verifică existența utilizatorului.
    2. Verifică să nu existe deja LectureAttendee pentru perechea user/lecture.
    3. Încarcă lecture; verifică Status == InProgress.
    4. Verifică faptul că utilizatorul nu este deja alăturat unui alt curs InProgress.
    5. Verifică faptul că utilizatorul este Student.
    6. Apelează lecture.Location.Value.DistanceTo(Location.FromString(command.Position)).
    7. Dacă distanța > 100 metri, aruncă ValidationException.
    8. Creează LectureAttendee(lectureId, userId, locationAtJoin), persistă.
    9. Returnează Result.Ok().

  5.4.5  SubmitQuiz
  ------------------
  Comandă:  SubmitQuizCommand(UserId, QuizLectureId)
  Handler:  SubmitQuizCommandHandler
    1. Verifică să nu existe UserSubmission pentru această pereche user/quiz-lecture.
    2. Încarcă QuizLecture și confirmă că este încă activ.
    3. Încarcă toate UserAnswers pentru acest user/quiz-lecture.
    4. Încarcă Quiz-ul cu toate întrebările și opțiunile.
    5. Pentru fiecare întrebare, însumează punctele dacă utilizatorul a marcat opțiunea corectă.
    6. Calculează MaxScore ca suma tuturor punctelor întrebărilor.
    7. Creează UserSubmission.Create(userId, quizLectureId, now, score, max).
    8. Persistă și returnează Result.Ok().

5.5  Common/Results – Monad-ul Result
--------------------------------------
Stratul Application introduce două tipuri generice de programare orientată pe traseu
(railway-oriented) care înlocuiesc excepțiile pentru controlul fluxului între straturi:

  Result<T>  – învelește o valoare de succes de tip T sau o listă de înregistrări Error.
  Result     – versiune non-generică pentru operații care nu returnează valoare.

  Ambele sunt readonly structs (tipuri valoare) pentru performanță (fără alocare pe heap
  pe traseele de succes).

  Metode factory pentru Result<T>:
    Result<T>.Ok(T value)              – HTTP 200
    Result<T>.Created(T value)         – HTTP 201
    Result<T>.NoContent()              – HTTP 204
    Result<T>.OkNullable(T? value)     – OK cu valoare posibil null
    Result<T>.Fail(Error error)        – eșec cu o singură eroare
    Result<T>.Fail(params Error[])     – eșec cu mai multe erori

  Combinatori funcționali pe Result<T>:
    Map<TResult>(Func<T, TResult>)     – transformă valoarea de succes
    Bind<TResult>(Func<T, Result<TResult>>) – înlănțuie operații fallibile
    Match<TResult>(onSuccess, onFailure)    – pattern-match pe stare
    Tap(Action<T>)                     – efect secundar la succes, returnează self
    TapFailure(Action<IReadOnlyList<Error>>) – efect secundar la eșec, returnează self
    TryGetValue(out T)                 – model bool + out

  Înregistrare Error:
    sealed record Error(Code, Message, ErrorKind, Metadata?)
    Factory: Error.Validation, Error.NotFound, Error.Conflict, Error.Unspecified

  Enumerarea ErrorKind:
    Failure(0), Validation(1), NotFound(2), Conflict(3), Unauthorized(4),
    Forbidden(5)

  Enumerarea SuccessKind:
    Ok, Created, NoContent

5.6  MediatR Pipeline Behaviors
--------------------------------

  5.6.1  ExceptionToResultBehavior<TRequest, TResponse>
  -------------------------------------------------------
  Împachetează execuția fiecărui handler într-un try/catch. Tabelul de mapare a excepțiilor:

    DomainException          → Error.Validation(message)       [fără log]
    ValidationException      → Error.Validation(message)       [fără log]
    FluentValidation.
      ValidationException    → Error.Validation(message)       [fără log]
    KeyNotFoundException      → Error.NotFound(message)        [Warning]
    Orice altă excepție      → Error.Unspecified("unexpected") [Error]

  După mapare, verifică dacă TResponse este Result sau Result<T>; dacă da, apelează
  factory-ul static Fail prin reflection și returnează eșecul fără re-aruncare.
  Acest lucru asigură că handler-ele nu propagă excepții către controller-e.

  5.6.2  ValidationBehavior<TRequest, TResponse>
  ------------------------------------------------
  Înainte de invocarea handler-ului, colectează toate instanțele IValidator<TRequest> din
  DI, le rulează concurent (Task.WhenAll) și aruncă un FluentValidation.
  ValidationException care agregă toate eșecurile. Acesta este apoi prins de
  ExceptionToResultBehavior de mai sus, convertindu-se într-un răspuns HTTP 400.

5.7  Utilitare JWT (Common/Jwt/)
---------------------------------
  JwtTokenGenerator.GenerateToken(userId, email, userType, expiresInMinutes=60)
    Construiește un JWT semnat HS256 cu claim-uri:
      sub  = userId (string)
      email = email utilizator
      role  = userType.ToString() (ex. "Student", "Professor")
      jti   = Guid.NewGuid().ToString()
    Semnează cu variabila de mediu JWT_SECRET_KEY.

  ClaimsPrincipalExtensions
    Metode de extensie pe ClaimsPrincipal:
      Guid GetUserId()     – citește claim-ul sub sau NameIdentifier
      UserType GetUserType() – citește claim-ul role, parsează în enum UserType
    Ambele aruncă UnauthorizedAccessException dacă claim-ul lipsește sau este invalid.

5.8  Hashing-ul parolelor (Common/Hash/PasswordHasher.cs)
-----------------------------------------------------------
  PasswordHasher este un utilitar static care folosește Argon2id din
  Konscious.Security.Cryptography. Parametri:
    SaltSize:             16 bytes (aleator, generat per parolă)
    HashSize:             32 bytes
    Iterations:           3
    MemorySize:           131,072 KB = 128 MB
    DegreeOfParallelism:  2

  HashPasswordAsync(password) → Base64(salt[16] || hash[32])
  VerifyPasswordAsync(password, storedBase64) → bool

  Verificarea folosește o buclă XOR de comparație în timp constant (ConstantTimeEquals)
  pentru a preveni atacurile side-channel bazate pe timing.

    – PK: Id (nu este generat de DB; aplicația atribuie Guid)
    – UserType stocat ca string (max 32)
    – Name max 100, Email max 255, Password max 255
    – Index unic pe Email

  LectureConfiguration
    – Tabel: "lectures"
    – PK: Id (nu este generat de DB)
    – Status stocat ca string (max 32)
    – StartTime și Duration obligatorii
    – Location mapat prin LocationConverter (VARCHAR 256 nullable)
    – HasMany(Attendees) cu ștergere în cascadă; navigare bazată pe câmp

  LectureAttendeeConfiguration
    – Tabel: "lecture_attendees"
    – PK compus: { LectureId, UserId }
    – FK către Lecture (cascade delete) și către User (restrict)
    – Index pe UserId
    – LocationAtJoin mapat prin LocationConverter (obligatoriu)

  UserFollowingConfiguration
    – Tabel: "user_followings"
    – PK compus: { FollowedId, FollowerId }
    – FK către User pe FollowerId (Restrict) și FollowedId (Restrict)

  QuizConfiguration
    – Tabel: "quizzes"; HasMany(Questions) cu cascade delete; navigare bazată pe câmp.

  QuizQuestionConfiguration
    – Tabel: "quiz_questions"; Points cu precizie (10, 2); HasMany(Options).

  QuizOptionConfiguration
    – Tabel: "quiz_options"; Text max 512.

  QuizLectureConfiguration
    – Tabel: "quiz_lectures"; index compus pe { LectureId, EndTimeUtc }
      pentru căutare eficientă a quiz-ului activ.

  UserAnswerConfiguration
    – Tabel: "user_answers"
    – Index unic pe { UserId, QuizLectureId, QuestionId, OptionId }
      (previne înregistrări duplicate de răspuns)
    – Index suplimentar pe { UserId, QuizLectureId } (recuperare rapidă a răspunsurilor)

  UserSubmissionConfiguration
    – Tabel: "user_submissions"
    – Score și MaxScore cu precizie (18, 2)
    – Index unic pe { UserId, QuizLectureId } (o trimitere per student)

6.4  Convertoare de valori  (Persistence/Converters/)
------------------------------------------------------
  LocationConverter  (ValueConverter<Location, string>)
    Convertește obiectul de valoare Location către/dinspre o singură coloană de bază
    de date de tip VARCHAR(256), cu formatul "lat,lon,accuracy" în InvariantCulture.
    Acest lucru evită necesitatea unui tabel GPS separat sau a tipurilor PostgreSQL geometry,
    menținând schema simplă și păstrând în același timp tipul de valoare bogat din domeniu.

6.5  Repository generic  (Repositories/GenericRepository.cs)
--------------------------------------------------------------
  O clasă de bază generică deschisă GenericRepository<T> unde T : Entity<Guid>
  implementează IRepository<T> folosind EF Core:

    AddAsync    → _dbSet.AddAsync
    Delete      → _dbSet.Remove
    DeleteByIdAsync → FindAsync apoi Remove (aruncă KeyNotFoundException dacă lipsește)
    GetAllAsync → AsNoTracking().ToListAsync
    GetByIdAsync → FirstOrDefaultAsync pe Id
    SaveChangesAsync → _db.SaveChangesAsync
    Update      → _dbSet.Update

  Toate repository-urile specializate moștenesc această clasă, suprascriind sau
  extinzând unde este necesar.

7.3  Mapare Result-HTTP  (Common/ErrorMapper.cs)
--------------------------------------------------
  Clasa statică ResultActionResultExtensions oferă metoda de extensie
  ToActionResult folosită de toate controller-ele pentru a converti Result/Result<T>
  în IActionResult:

  Mapare succes (SuccessKind → status HTTP):
    Ok       → 200 OK (cu corp de valoare)
    Created  → 201 Created (cu corp de valoare)
    NoContent→ 204 No Content

  Mapare eșec (ErrorKind → status HTTP):
    Validation   → 400 Bad Request
    NotFound     → 404 Not Found
    Conflict     → 409 Conflict
    Unauthorized → 401 Unauthorized
    Forbidden    → 403 Forbidden
    Failure      → 500 Internal Server Error

  Când sunt returnate mai multe erori, eroarea "cea mai semnificativă"
  (determinată printr-un clasament de prioritate: Validation > NotFound > Conflict > Unauthorized > Forbidden
  > Failure) este folosită ca status HTTP și titlu principal. Toate erorile sunt
  incluse în corpul răspunsului sub o extensie ProblemDetails numită "errors".

CAPITOLUL 9 – AUTENTIFICARE ȘI SECURITATE
================================================================================

9.1  Fluxul de autentificare
-----------------------------
1. Clientul trimite credentiale prin POST la POST /api/users/login.
2. LoginUserCommandHandler verifică email-ul și hash-ul parolei Argon2id.
3. La succes, JwtTokenGenerator.GenerateToken produce un JWT semnat HS256
   care conține sub (userId), email, role (UserType) și jti (ID unic token).
4. Controller-ul API atașează token-ul ca cookie HttpOnly, Secure, SameSite=Lax
   numit "AttendanceApp.Jwt", cu expirare la o oră.
5. Cererile ulterioare din browser includ automat acest cookie.
6. JwtBearerEvents.OnMessageReceived citește valoarea cookie-ului și setează
   context.Token astfel încât middleware-ul standard JWT bearer să îl valideze.
7. Atributele [Authorize] pe acțiuni/clase de controller declanșează 401 dacă
   token-ul lipsește sau este invalid.

9.2  Modelul de autorizare
---------------------------
Aplicația folosește autorizare bazată pe rol prin claim-uri UserType. Autorizarea
la nivel de reguli de business (de ex. "doar profesorul care a creat cursul îl
poate porni") este impusă în interiorul handler-elor Command, nu prin middleware-ul
de politici ASP.NET Core. Handler-ele verifică ID-ul sau tipul utilizatorului
autentificat în raport cu resursa înainte de a efectua mutații.

9.3  Securitatea parolelor
---------------------------
  – Argon2id cu 128 MB memorie, 3 iterații, paralelism 2.
  – Salt aleator de 16 bytes per parolă; asigură hash diferit pentru parole identice.
  – Comparație în timp constant în timpul verificării.
  – Parola este stocată doar ca hash; textul în clar nu este niciodată persisat.

9.4  Securitatea token-ului
----------------------------
  – Cookie HttpOnly: inaccesibil din JavaScript; reduce riscul XSS.
  – Flag Secure: transmis doar prin HTTPS.
  – SameSite=Lax: permite navigare top-level cross-site, dar nu cereri
    state-changing vulnerabile la CSRF din site-uri terțe.
  – JWT are ClockSkew = TimeSpan.Zero pentru impunerea precisă a expirării.
  – Secretul este încărcat din variabilă de mediu; nu este hardcodat în codul
    sursă la nivel de producție (există doar un fallback de dezvoltare locală).

9.5  Navigare în browser
-------------------------
  Când o cerere browser neautentificată (Accept: text/html) ajunge la un endpoint
  protejat, evenimentul OnChallenge redirecționează către /auth/login în loc să
  returneze HTTP 401, oferind o experiență de login fluidă. Cererile AJAX primesc
  în continuare HTTP 401.

CAPITOLUL 12 – PROIECTAREA BAZEI DE DATE
================================================================================

12.1  Prezentare generală Entity-Relationship
----------------------------------------------
Baza de date PostgreSQL conține zece tabele. Cheile primare sunt GUID-uri (UUID-uri)
generate de aplicație pentru a evita expunerea ID-urilor secvențiale și pentru a
susține generarea distribuită.

  users
    id               uuid  PK
    type             varchar(32)  ["Student","Professor","Admin"]
    name             varchar(100) NOT NULL
    email            varchar(255) NOT NULL  UNIQUE
    password         varchar(255) NOT NULL

  lectures
    id               uuid  PK
    professor_id     uuid  NOT NULL  (fără constrângere FK declarată; doar logică)
    status           varchar(32)  ["Scheduled","InProgress","Ended","Canceled"]
    name             varchar(?)
    description      varchar(?)
    start_time       timestamp with time zone  NOT NULL
    duration         interval  NOT NULL
    location         varchar(256)  NULLABLE  ["lat,lon,accuracy"]

  lecture_attendees
    lecture_id       uuid  PK (parte din compus), FK → lectures ON DELETE CASCADE
    user_id          uuid  PK (parte din compus), FK → users ON DELETE RESTRICT
    time_joined      timestamp with time zone
    location_at_join varchar(256)  NOT NULL
    INDEX (user_id)

  user_followings
    followed_id      uuid  PK (parte din compus), FK → users ON DELETE RESTRICT
    follower_id      uuid  PK (parte din compus), FK → users ON DELETE RESTRICT

  quizzes
    id               uuid  PK
    name             varchar(256)  NOT NULL
    duration         interval  NOT NULL
    professor_id     uuid  NOT NULL
    created_at_utc   timestamp with time zone  NOT NULL

  quiz_questions
    id               uuid  PK
    quiz_id          uuid  NOT NULL  FK → quizzes ON DELETE CASCADE
    text             varchar(1024)  NOT NULL
    order            int  NOT NULL
    points           numeric(10,2)  NULLABLE

  quiz_options
    id               uuid  PK
    question_id      uuid  NOT NULL  FK → quiz_questions ON DELETE CASCADE
    text             varchar(512)  NOT NULL
    order            int  NOT NULL
    is_correct       bool  NOT NULL

  quiz_lectures
    id               uuid  PK
    lecture_id       uuid  NOT NULL
    quiz_id          uuid  NOT NULL
    created_at_utc   timestamp with time zone  NOT NULL
    end_time_utc     timestamp with time zone  NOT NULL
    INDEX (lecture_id, end_time_utc)

  user_answers
    id               uuid  PK
    user_id          uuid  NOT NULL
    quiz_lecture_id  uuid  NOT NULL
    question_id      uuid  NOT NULL
    option_id        uuid  NOT NULL
    choice           bool  NOT NULL
    UNIQUE (user_id, quiz_lecture_id, question_id, option_id)
    INDEX (user_id, quiz_lecture_id)

  user_submissions
    id               uuid  PK
    user_id          uuid  NOT NULL
    quiz_lecture_id  uuid  NOT NULL
    submitted        bool  NOT NULL
    submitted_at_utc timestamp with time zone  NOT NULL
    score            numeric(18,2)  NOT NULL
    max_score        numeric(18,2)  NOT NULL
    UNIQUE (user_id, quiz_lecture_id)

12.2  Decizii cheie de proiectare
----------------------------------
  – GUID-urile generate de aplicație (ValueGeneratedNever) împiedică DB-ul să
    atribuie ID-uri întregi secvențiale, care ar putea expune informații despre numărul de rânduri.
  – PK-urile compuse pe lecture_attendees și user_followings servesc drept
    constrângeri naturale de unicitate, prevenind duplicatele fără indici suplimentari.
  – Enum-urile de status și tip sunt stocate ca string-uri, nu ca întregi, pentru
    lizibilitate în interogările SQL și în migrații.
  – Location este stocat ca o singură coloană VARCHAR, nu ca geometrie PostGIS,
    menținând dependențele minime și permițând totodată calculul Haversine
    la nivelul aplicației.
  – Ștergere în cascadă din lectures → lecture_attendees (ștergerea unui curs
    șterge toate înregistrările de prezență). Fără cascadă pe users; înregistrările
    de prezență sunt păstrate dacă un cont utilizator este șters (restrict).
  – Indexul compus quiz_lectures pe (lecture_id, end_time_utc) susține interogarea
    "get active quiz for lecture" care filtrează după ambele coloane.
  – Indexul unic user_answers împiedică un student să înregistreze două răspunsuri
    diferite pentru aceeași opțiune în același quiz-lecture (toggle-ul trece prin
    un flux de tip upsert UpdateChoice).
  – Indexul unic user_submissions (user_id, quiz_lecture_id) impune o singură
    trimitere per student per quiz-lecture la nivelul bazei de date.

CAPITOLUL 13 – MODELE ȘI PRINCIPII DE PROIECTARE APLICATE
================================================================================

13.1  Clean Architecture / Hexagonal Architecture
--------------------------------------------------
Aplicația urmează strict Dependency Rule: straturile interioare nu au
referințe către straturile exterioare. Domain are zero dependențe externe.
Application depinde doar de Domain. Straturile de infrastructură și prezentare
depind de straturile interioare, dar nu unul de celălalt (exceptând composition
root-ul din Web).

13.2  Domain-Driven Design (DDD)
---------------------------------
Concepte DDD cheie aplicate:

  Entities – obiecte cu identitate persistentă (User, Lecture, LectureAttendee,
    Quiz, QuizQuestion, QuizOption, QuizLecture, UserAnswer, UserSubmission,
    UserFollowing).

  Aggregate Roots – proprietarii graniței de consistență: User, Lecture, Quiz.
    Codul extern poate modifica agregatele doar prin metodele lor publice; colecțiile
    interne sunt ascunse în spatele proprietăților read-only.

  Value Objects – Location este un readonly record struct care implementează
    egalitatea prin valoare structurală, nu prin identitate de referință. Nu are identitate
    (fără Id), este imuabil și încapsulează logica Haversine.

  Domain Exceptions – DomainException este semnalul exclusiv de eroare al domeniului;
    toate încălcările regulilor de business aruncă acest tip.

  Repository Interfaces – definite în Domain (ILectureRepository etc.) cu
    implementări în Infrastructure, decuplând logica de domeniu de persistență.

  Guard Clauses – clasa internă Guard impune invariantele cât mai devreme
    posibil (la construcția obiectului), făcând imposibile stările invalide.

  Factory Methods – Quiz.Create și QuizLecture.Create sunt metode factory
    statice care validează intrările înainte de construirea obiectului, prevenind
    existența agregatelor parțial inițializate.

13.3  Command Query Responsibility Segregation (CQRS)
------------------------------------------------------
Comenzile (mutații) și interogările (citiri) sunt separate strict în perechi
distincte request/handler MediatR. Acest lucru oferă claritate a intenției,
testabilitate independentă și opțiunea de a scala separat căile de citire și scriere
în viitor.

13.4  Modelul Repository
-------------------------
Tot accesul la date este abstractizat în spatele interfețelor repository.
Controller-ele și handler-ele depind de interfețe (ILectureRepository, IUserRepository etc.),
niciodată direct de DbContext. Acest lucru permite:
  – Înlocuire facilă în teste (InMemory EF înlocuiește Npgsql).
  – Responsabilitate unică: repository-urile încapsulează toată logica de interogare.
  – Compatibilitate TDD: handler-ele pot fi testate unitar pe repository-uri mock.

13.5  Modelul Pipeline / Decorator (MediatR Behaviors)
-------------------------------------------------------
Aspectele transversale (validare, mapare excepție-la-rezultat) sunt aplicate
uniform tuturor handler-elor prin pipeline-ul MediatR. Acest lucru menține handler-ele
individuale curate, fără boilerplate try/catch și invocare FluentValidation.
Ordinea comportamentelor contează:
  1. ExceptionToResultBehavior (cel mai exterior) prinde excepțiile aruncate de straturile interioare
  2. ValidationBehavior validează request-ul ÎNAINTE ca handler-ul să ruleze

13.6  Railway-Oriented Programming (Tipul Result)
--------------------------------------------------
Tipurile Result<T> / Result implementează o versiune ușoară de ROP, un model
de programare funcțională popularizat de Scott Wlaschin. În locul excepțiilor
pentru controlul fluxului, operațiile returnează o uniune discriminată de succes
sau eșec. Apelanții folosesc Match, Map, Bind, Tap pentru a înlănțui operații fără
verificări defensive de null. Metoda de extensie ToActionResult este interpretul final
care traduce semantica Result în semantică HTTP.

13.7  Modelul Value Converter (EF Core)
----------------------------------------
LocationConverter demonstrează modelul EF Core Value Converter pentru a
serializa un tip de valoare de domeniu bogat (Location struct) într-un tip simplu
de coloană în baza de date (VARCHAR). Acest lucru păstrează domeniul liber de
preocupări de persistență, stocând în același timp datele eficient.

13.8  Modelul Extension Method
-------------------------------
Metodele de extensie sunt folosite extensiv:
  – ClaimsPrincipalExtensions.GetUserId() / GetUserType() pentru extragere curată a claim-urilor.
  – ResultActionResultExtensions.ToActionResult() pentru mapare HTTP.
  – HttpContextExtensions.ReadAsAsync<T>() în helper-ele de test.

13.9  Modelul Options / Configurare
------------------------------------
Connection strings și secretele JWT sunt externalizate prin variabile de mediu
încărcate de DotNetEnv. Acest lucru urmează metodologia Twelve-Factor App, păstrând
configurația sensibilă în afara controlului versiunilor.

13.10  Geofencing cu distanța Haversine
----------------------------------------
Funcționalitatea de alăturare la prezență implementează un geofence de 100 metri
folosind formula Haversine integral în tipul de valoare Location din stratul Domain.
Acesta este un exemplu clar de logică de domeniu care nu se scurge în straturile
infrastructure sau application—calculul distanței este o funcție pură pe două valori de locație.

CAPITOLUL 14 – REZUMAT ȘI LUCRĂRI VIITOARE
================================================================================

14.1  Rezumat
--------------
AttendanceApp este o aplicație web full-stack cuprinzătoare care implementează o
platformă de prezență la curs bazată pe GPS și quiz-uri în timpul cursului.
Proiectul demonstrează aplicarea unor practici mature de inginerie software:

  – Clean Architecture cu separare strictă pe straturi.
  – Domain-Driven Design cu aggregate roots bogate, obiecte de valoare și
    clauze guard care fac imposibilă reprezentarea stărilor invalide de domeniu.
  – CQRS prin MediatR, organizând peste 40 de use-case-uri în unități discrete și testabile.
  – Railway-Oriented Programming printr-un monad custom Result<T>.
  – MediatR pipeline behaviors pentru validare transversală și tratarea erorilor.
  – Hashing de parole Argon2id conform standardelor OWASP.
  – Autentificare cu cookie JWT HTTP-only rezistentă la XSS.
  – Prezență cu geofencing folosind formula Haversine implementată în domeniu.
  – EF Core Code-First cu configurații Fluent API, convertoare de valori și un
    istoric curat al migrațiilor.
  – Front-end MVC randat pe server cu module JavaScript per view.
  – Acoperire de test cuprinzătoare: 10 clase de test pentru entități de domeniu, 1 clasă
    pentru tipul result, 1 clasă de test pentru maparea API și 9 clase de test de integrare
    pentru controller-e care acoperă principalele comportamente ale endpoint-urilor HTTP.
  – Infrastructură containerizată folosind Docker Compose pentru pornire cu o singură comandă.

14.2  Posibile îmbunătățiri viitoare
-------------------------------------
  1. Notificări în timp real prin WebSockets sau SignalR
       În locul polling-ului, un hub în timp real ar putea transmite direct către
       studenții conectați schimbările de stare ale cursului (Started, Ended) și
       activările de quiz.

  2. Integrare OAuth2 / OpenID Connect
       Social login (Google, Microsoft) pentru studenți și SSO cu furnizorii
       de identitate ai universității prin OIDC.

  3. Refresh Tokens
       Implementarea JWT actuală folosește un singur token de o oră. O schemă
       cu refresh token ar permite persistența sesiunilor fără re-login.

  4. Control de acces bazat pe roluri, granular fin
       Un sistem de autorizare bazat pe politici ar putea înlocui verificările din handler,
       permițând securitate declarativă la nivel de atribut.

  5. Dashboard Admin
       Rolul UserType.Admin există în domeniu, dar nu au fost implementate UI sau
       endpoint-uri specifice de administrare.

  6. Încărcări media
       Permiterea profesorilor să atașeze materiale (slide-uri, PDF-uri) la cursuri,
       stocate în cloud blob storage cu URL-uri persistate în înregistrarea cursului.

  7. Analitice și raportare
       Statistici agregate de prezență per profesor / curs / interval de date
       exportate ca CSV sau vizualizate în aplicație.

  8. Aplicație mobilă
       O aplicație nativă iOS/Android care utilizează senzorul GPS nativ (acuratețe
       mai bună decât browser Geolocation API) pentru prezență și participare la quiz.

  9. Extensia geospațială PostGIS
       Pentru interogări de proximitate mai complexe (ex. verificări la limitele campusului),
       înlocuirea coloanei VARCHAR location cu tipuri PostGIS geometry ar permite
       interogări spațiale pe server.

  10. Event Sourcing pentru audit trail de prezență
       Înregistrarea fiecărui eveniment de prezență (join, leave, re-join) cu timestamp-uri
       pentru a oferi un jurnal complet de audit pentru rezolvarea disputelor.
